# 编排流程与工作包交接规范

> 详述从接活到交付的完整编排，含 Gate 清单、工作包规范、状态机。

---

     ▼

### Step 5.5 标题工程优化（标题工程师 08 接管）

**触发**：Gate A 初稿确认后自动进入。
**承载角色**：08-title-engineer（标题工程师）

**执行内容**：
1. 读取初稿核心观点 + 选题卡片
2. 生成 8-10 个候选标题（含打开率预估评分）
3. 按平台差异化输出（公众号 22-30 字 / 知乎 15-25 字 / 小红书 15-20 字）
4. 生成公众号摘要（15-30 字，3 个候选）
5. 输出封面图建议

**Gate A-Title · 标题确认**：
```
⛔ 必须输出：
- 候选标题列表（含评分）
- 公众号摘要候选
- 封面图建议
⏸️ 用户选定标题后，方可进入 Step 6 审查。
```

**衔接说明（WorkBuddy 模式）**：
- 输入：用户粘贴初稿核心观点 + 选题卡片（来自 01-topic-scout 和 04-writer 的输出）
- 输出：用户选定的标题 + 摘要 → 手动传递给 05-reviewer 和 07-formatter

### Step 9 运营策略注入（运营策划师 10 接管）

**触发**：Gate E 终稿确认后、排版前执行。
**承载角色**：10-ops-strategist（运营策划师）

**执行内容**：
1. 系列化标记：本文是否属于某系列？下篇预告？
2. CTA 设计：根据文章类型选择 CTA 组合（关注/在看/转发/评论）
3. 评论区置顶话术设计
4. 发布时间建议
5. 知乎问题匹配（如目标平台含知乎）：推荐适合回答的高流量问题

**输出格式**：
```
📋 运营策略包
- 系列标记：{属于系列 XX 第 N 篇 / 独立文章}
- CTA 组合：关注({话术}) + 在看({话术}) + 评论({话术})
- 评论区置顶：{话术}
- 建议发布时间：{时段 + 理由}
- 知乎匹配问题：{问题标题}（热度/竞争度/匹配度）
```

**衔接说明（WorkBuddy 模式）**：
- 输入：用户粘贴终稿标题 + 文章类型 + 目标平台（来自 05-reviewer 的终稿）
- 输出：运营策略包 → 手动传递给 07-formatter 排版时整合

### Step 13 发布后数据闭环（数据分析师 09 接管）

**触发**：文章发布后 72 小时（用户主动触发，提供发布后数据）。
**承载角色**：09-data-analyst（数据分析师）

**执行内容**：
1. 数据采集：用户提供公众号/知乎后台数据（截图或手动输入）
2. 表现归因分析：选题/标题/内容/时间四维归因
3. 与历史数据对比（如有）
4. 策略迭代建议：
   - 选题方向调整建议 → 反馈给 01-topic-scout
   - 标题策略优化建议 → 反馈给 08-title-engineer
   - 内容深度调整建议 → 反馈给 04-writer
   - 发布时间优化建议 → 反馈给 10-ops-strategist

**闭环机制**：
- 每篇文章产出「表现报告」
- 每月产出「月度策略迭代报告」
- 关键发现写入风格库或选题策略

**衔接说明（WorkBuddy 模式）**：
- 输入：用户提供发布后数据（阅读量/互动数据/涨粉数据）
- 输出：表现报告 + 策略建议 → 用户在下次选题/写作时参考

## Gate 清单速查

### Gate 命名规范

| 编号 | 名称 | 对应阶段 | 说明 |
|---|---|---|---|
| Gate 0 | 需求共识 | Step 1 后 | 需求确认，HKRT 通过后 |
| Gate A1 | 素材确认 | Step 3 后 | 动笔前素材清单确认 |
| Gate A2 | 骨架拍板 | Step 3.5 后 | 仅深度模式，横纵骨架确认 |
| Gate A | 初稿确认 | Step 5 后 | 初稿 + 字数确认 |
| Gate A-Title | 标题确认 | Step 5.5 后 | 用户选定标题 + 摘要确认 |
| Gate B | 审查确认 | Step 6 后 | 审查报告确认 |
| Gate C | 去 AI 味确认 | Step 7 后 | 处理摘要确认 |
| Gate E | 终稿确认 | Step 8 后 | 终稿 + 引用台账确认 |

> 命名由来：数字 Gate（0）表示需求阶段，字母 A 表示写作前准备，字母 B/C 表示写作后审查，字母 E 表示最终交付。A1/A2 是 A 的子门禁。

### Gate 通过条件速查

| Gate | 阶段 | 把关人 | 必须输出 | 通过条件 |
|---|---|---|---|---|
| Gate 0 | 需求共识 | 用户 | 需求共识摘要 + HKRT 评级 | HKRT ≥及格 + 用户确认 |
| Gate A1 | 素材确认 | 用户 | 素材清单（四维） | 用户确认 |
| Gate A2 | 骨架拍板 | 用户 | 横纵骨架（仅深度模式） | 用户拍板 |
| Gate A | 初稿确认 | 用户 | 初稿链接 + 字数 | 检查清单全 ✅ + 用户回"继续" |
| Gate A-Title | 标题确认 | 用户 | 候选标题列表（含评分） + 摘要候选 + 封面图建议 | 用户选定标题 |
| Gate B | 审查确认 | 用户 | 完整审查报告 L1-L4 | 报告完整 + 用户确认 |
| Gate C | 去 AI 味确认 | 用户 | 处理摘要（高频/中频/低频） | 摘要完整 + 用户确认 |
| Gate E | 终稿确认 | 用户 | 终稿链接 + 引用台账 | 检查清单全 ✅ + 用户回"继续" |

**Gate B 打回协议**：

当 Gate B 审查不通过时，执行以下打回流程：

1. **打回触发条件**：L1 硬规则命中 / L4 活人感终审不通过 / 用户主动要求返工
2. **打回产物**：审查员输出「打回摘要」——包含具体不合格段落编号、问题分类（硬规则/风格/质量/活人感）、修复建议
3. **打回路径**：审查员 → 撰稿人（Step 4 重新加载），只重做最小可重做单元（单段/单节），不全文重写
4. **重做上限**：同一段落打回 ≤ 3 次；超过 3 次升级为人工判断（预算护栏触发）
5. **打回后准出**：撰稿人修复后重新走 Gate A（初稿确认） → Gate B（审查确认），不跳过任何 Gate

## 三层 QA 触发点速查（吴恩达 Loop Engineering 框架）

完整 QA 框架见 [qa-framework.md](qa-framework.md) v1.1（含 13 个方法论吸收）。Gate 体系映射三层 QA：

| Gate | 第一层 QA（执行·挑错） | 第二层 QA（读者价值·PM） | 第三层 QA（长期竞争力·战略） |
|---|---|---|---|
| Gate 0 需求共识 | — | 读者画像 + 价值承诺（秘书，吸收 8 个性化评判） | — |
| Gate A1 素材确认 | 事实核查 + 引用源（审查员，吸收 2 FActScore + 吸收 3 CRITIC 外验） | — | — |
| Gate A2 骨架拍板 | — | — | — |
| Gate A 初稿确认 | — | 读者价值验证 + 体验曲线 + 信息密度（审查员 L3 升级，吸收 5/6/7） | — |
| Gate B 审查确认 | L1-L4 四层金字塔（审查员，吸收 1 偏见消除 + 吸收 3 内省） | — | — |
| Gate C 去 AI 味 | 三重保障（审查员） | — | — |
| Gate E 终稿确认 | 引用台账（审查员） | 平台适配验证（秘书 + 排版师） | 辨识度 + 认知资产 + 差异化（风格官，吸收 11/12） |
| Gate E 后 | — | — | 上下文资产沉淀（风格官）+ 踩坑反思（审查员+撰稿人，吸收 9 Reflexion）+ Agent 自评（三角色，吸收 10）+ 指纹漂移检测（风格官，吸收 11） |
| 季度复盘 | — | — | 长期壁垒评估 + 画像命中率 + 认知投降指标（风格官 + 作者，吸收 13） |

**承载原则**（v1.1 扩展）：不新增角色。第一层由审查员承载（已有，吸收 1-4 强化执行准则）；第二层秘书 + 审查员 L3 升级（吸收 5-8 提供方法论支撑）；第三层风格官 + 作者本人（吸收 9-13 把定性检查升级为可度量机制）。对应吴恩达「角色融合与边界重构」洞察。13 个方法论吸收全部以「方法论参考」形式集成，不直接下载代码——详见 [external-skills.md](external-skills.md) §十。

## 防跳协议（最高优先级）

1. 每步输出前必须声明 `[当前: Step X · 下一步: Step Y（等待用户确认）]`
2. 遇 Gate 标记，输出 Gate 内容后立即停止，用户未明确回复前禁止输出下一步任何内容
3. 追问/质检阶段：精确度未达标禁止跳过，检查清单有 ❌ 禁止推进
4. 当前阶段未通过 Gate，不得加载下一阶段子文件
5. 违反 = 任务失败，从当前步骤重新开始

## 工作包交接规范

任务从一个 Agent 转给下一个时，生成带版本、可校验的工作包：

```
📦 工作包 v{N} · {from} → {to}
─── 目标 ───
真正想做成什么 / 当前目标 / 限制 / 验收标准
─── 事实与证据 ───
确定的事实 / 假设 / 不确定或冲突（各标可信度）
证据和产物放在哪里（按引用取回，不重复传大文件）
─── 决定 ───
已做过哪些决定 / 哪些内容被省略及如何找回
─── 交接印记 ───
交给谁 / 对方是否核对 / 是否要求补充信息
─── 主旨锁 ───
「主旨锁: {一句话复述素材核心主旨与情绪基调}」
─── 素材契约 ───
素材保留率 ≥70% / 替换需声明 / AI 占比上限
```

**硬约束**：
- AI 内心独白、密码、密钥、登录令牌一概不进工作包
- 大文件只存一份，靠引用传递
- 接收方必须明确点「收到 / 要补充信息 / 退回」
- 发送方不算自己干完了，直到接收方点「收到」

## 状态机（后台控制系统说了算）

AI 只能建议下一步，只有调度系统能在 8 种状态间切换：
继续 / 改计划 / 重试 / 等人 / 完成 / 失败 / 取消 / 预算用尽

每次干活留痕：稳定编号、当前在四步的哪一步、产物、质检、卡点、可回看事件记录、检查点和恢复点、用量和预算。

每个副作用必须可重复、可去重或显式确认；禁止整段重来，只重做最小可重做单元。

---

## 预算护栏

每次干活设上限：循环次数、重试、时长、工具调用、模型调用、花费。进度停滞被发现，不无限自我反思、不烧钱。

改数字员工、改技能、改规则、改工作流不算普通任务循环——必须创建用户可见的迭代提案，由用户确认。

## 深度模式开关

**触发**：用户在 Step 1/2 说过「写深一点」「横纵分析」「深度文」「5000-10000 字」等。
**适用类型**：insight / review / lab（case、tutorial 可选；短文 <2500 字不开启）。

开启后变化：
1. 字数 3500-6000 → 5000-10000
2. 素材升级四维（+横纵专项必选 +实时热点必选）
3. 新增 Step 3.5 横纵骨架
4. 风格保持人味，咨询报告腔被 kenny-writer 消解
5. L1-L4 自检照常

输出模式判断（必做）：
```
🔧 模式判断
- 深度模式：开启 ✅ / 关闭 ❌
- 触发依据：{用户原话或"未命中触发词"}
- 字数预期：{3500-6000 或 5000-10000}
- 是否启用横纵骨架：{是/否}
- 是否启用实时热点检索：{是/否}
```
⛔ 模式判断未输出 → 不得加载 phase2。

## 关键动作人工把关（11 类）

这些动作 AI 不能自己拍板，必须人工判断或专业复核：
对外发送、公开发布、上线部署、删除、覆盖、付款、数据导出、记入长期记忆、晋升为公司知识、给出受监管的专业建议、最终交付。

按 5 类风险（受监管专业领域、不可逆或对外动作、知识晋升、权限范围不清、证据缺失或冲突）自动评估风险分，超阈值（默认 0.65）暂停请人确认。
