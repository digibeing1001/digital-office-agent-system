# QA 体系框架 · 三层 Loop 与上下文资产沉淀

> 本文件是 Agent Team 开展多类任务时的关键参考文档。
> 依据吴恩达 Loop Engineering 三层逻辑框架（原始文章：[Three Key Loops for Building Great Software](https://www.deeplearning.ai/the-batch/three-key-loops-for-building-great-software/)，The Batch, 2026-06-26），把 QA 从「挑错型」升级为「价值定义型 + 长期上下文资产沉淀型」。
> 不新增角色，是现有 8 角色（尤其审查员 / 秘书 / 风格官）的能力边界延伸——对应吴恩达「角色融合与边界重构」洞察。

## 〇、概念修正（基于吴恩达原文，v1.1 升级）

v1.0 基于「吴恩达三层 Loop」类比设计三层 QA，v1.1 根据[吴恩达原文](https://www.deeplearning.ai/the-batch/three-key-loops-for-building-great-software/)做 4 处概念修正：

### 修正 1：QA 横切三层 Loop，而非 QA = Loop

**原理解**：三层 QA = 三层 Loop 的等价物。
**修正**：吴恩达的三层 Loop 是**产品开发流程的反馈循环**（Agentic Coding Loop 分钟级 / Developer Feedback Loop 小时级 / External Feedback Loop 天周级），不是 QA 分层。准确表述是：**QA 横切三层 Loop，每层 Loop 都有自己的质量门**。

| 吴恩达三层 Loop | 我们的三层 QA | 角色定位 |
|---|---|---|
| Agentic Coding Loop（分钟级） | 第一层 · 执行层 | = Loop 的 Evals，负责「文章是否满足 spec」 |
| Developer Feedback Loop（小时级） | 第二层 · 读者价值层 | = Loop 中人的判断，负责「spec 是否对、读者价值是否成立」 |
| External Feedback Loop（天/周级） | 第三层 · 长期竞争力层 | = Loop 中市场反馈，负责「方向是否对、辨识度是否沉淀」 |

### 修正 2：「品味 = 上下文优势」是可工程化替换，不是品味换说法

**原理解**：品味在 AI 时代依然重要，只是换了说法。
**修正**：吴恩达原话——「**I prefer to think of it as humans having a context advantage, since that gives us a clearer path to helping AI systems get better.**」他用「上下文优势」**替换**「品味」概念，因为品味暗示不可传授，上下文优势指向可操作路径（把独有上下文显性化，AI 就能变好）。

**对我们的意义**：第三层 QA 不应只评估「品味好坏」，而应评估**「上下文是否被显性化沉淀」**——即作者独有的信息是否被持续写入 Voice Profile / 风格库 / 选题库 / 案例库。

### 修正 3：第三层反馈传导路径有多跳，所以慢，所以决定方向

**原理解**：第三层最慢但决定产品生死。
**修正**：吴恩达原话——「**This data informs the developer vision, which in turn continues to drive the detailed product spec, which in turn drives the coding agent.**」外层反馈不直接改代码，而是先修正人的判断，再传导到 spec，再到 agent。

**对我们的意义**：长期竞争力层 QA 的反馈（读者留存 / 辨识度数据）不能直接改下一篇文章，而要先回到「作者愿景」层（我要成为什么样的作者），再传导到选题 spec，再到写作 agent。**反馈路径长，所以更要重视。**

### 修正 4：读者价值层必须「同时做两件事」

**原理解**：读者价值层只做验证。
**修正**：吴恩达原话——「**It is important to do both!**」既要 bridge vision↔spec（构建），又要 get user feedback（验证）。

**对我们的意义**：读者价值层 QA 不能只做「验证读者反馈」，还要做「构建」——即把读者洞察持续翻译回下一篇的 spec。**只验证不构建 = 空想；只构建不验证 = 自嗨。**

## 一、为什么需要这份 QA 框架

当前 Agent Team 已有完善的「执行层 QA」：审查员四层金字塔审查 + 去 AI 味三重保障 + Gate 防跳协议 + 事实核查 + 引用源台账。这套体系能保证文章「没有错」，但停留在吴恩达所说的**第一层 Loop**（Agentic Coding Loop，分钟级自审迭代）。

吴恩达的颠覆性观点：第一层 Loop 是当前行业落地最成熟的基础层，但**恰恰是最不重要的一层**。真正的核心竞争力在第二层（Developer Feedback Loop，人介入判断 spec 是否对）和第三层（External Feedback Loop，市场反馈决定方向）。

**升级方向**：QA 不再只是审查员一个人的事，而是横切三层 Loop 的质量保障工作，每层在不同 Gate 触发，由不同角色承载。

## 二、三层 QA 框架

### 第一层 QA · 执行层（挑错型，审查员承载）

**目标**：确保文章「没有错」——事实准、引用全、无 AI 味、合规范。

**触发节点**：Gate A1（素材确认）/ Gate B（审查确认）/ Gate C（去 AI 味确认）

**已有检查项**（无需新增）：
- 事实核查：多信源交叉验证（重要事实 ≥2 个独立信源），空泛工具名检查
- 引用源完整性：`[SXX]` 编号连续，无编号不可写入研究简报
- 四层金字塔审查：L1 硬规则 / L2 风格一致性 / L3 内容质量 / L4 活人感终审
- 去 AI 味三重保障：作者站位复核 + humanize-chinese-writing 五层检查 + humanizer 33 模式中文适配表
- 卡兹克禁用清单扫描

**v1.1 新增方法论吸收**（来自 arxiv 论文 + GitHub 工具调研，不新增检查项，作为审查员执行准则）：

#### 吸收 1：LLM-as-judge 三类偏见消除（arxiv: 2306.05685, 2305.17926）

LLM 作为评判者存在三类系统性偏见，写作 QA 必须显式消除：

| 偏见类型 | 表现 | 写作场景的危害 | 消除方法 |
|---|---|---|---|
| **位置偏见** | 评判者偏好特定位置的答案 | 多版本对比时系统性偏好某一版本 | 交换顺序后多次取均值 |
| **冗长偏见** | 评判者偏好更长文本 | 新媒体读者偏好精炼，审查员会放行冗长、打回精炼 | 引入长度惩罚项；精炼优先 |
| **自我偏好偏见** | 评判者偏好同源模型输出 | 写作 Agent 和审查 Agent 同源时纵容自身缺陷 | **引入异质审查者**；关键维度人工抽检 |

**硬约束**：写作 Agent 与审查 Agent 必须是不同 instance / 不同 prompt / 最好不同模型。若同源，关键维度（事实/风格/价值）必须人工抽检——这是 Addy Osmani 说的「验证者 ≠ 执行者」红线，否则形成 confirmation bias 循环。

#### 吸收 2：FActScore 原子化事实核查（arxiv: 2305.14251）

**原方法**：把长文本拆解为原子事实序列，逐条验证是否被可靠知识源支持，计算支持率。

**写作场景适配**：审查员不应笼统说「这篇文章有问题」，应把文章拆成可验证的原子命题（每个论点、每个数据、每个引用），逐一标记：
- ✅ 已验证（≥2 个独立信源支持）
- ⚠️ 单一信源（需标注）
- 🔴 错误/存疑（必须修正或删除）

**与现有引用源台账的关系**：引用源台账（`[SXX]` 编号）已是原子化的基础，FActScore 补充了「逐条验证 + 支持率计算」的执行准则——不只记编号，还要标注每条的支持状态。

#### 吸收 3：CRITIC 工具增强审查（arxiv: 2305.11738）

**原方法**：LLM 生成输出后调用外部工具（搜索引擎查证事实、代码解释器验证逻辑）来评估特定方面。

**写作场景适配**：事实类声明用搜索工具核查（不靠模型脑补），数据类声明用计算工具验证，逻辑类推理用代码验证。

**审查员分两路执行**：
- **内省审查**（Self-Refine 式，arxiv: 2303.17651）：查语法 / 结构 / 表达 / 风格 / AI 味——模型自反馈快速迭代
- **外验审查**（CRITIC 式）：查事实 / 数据 / 引用——借助工具查证，不靠模型「脑补」

两者输出合并为原子化错误清单，每条标注：位置、类型、严重度、修复建议、可追溯证据。

#### 吸收 4：Evals 可演进（吴恩达原话 + Promptfoo 方法论）

**吴恩达原话**：「**If you find that the system repeatedly runs into certain problems, building a set of evals for the agent becomes useful.**」——反复踩坑的场景要固化为 evals。

**写作场景适配**：执行层 QA 不是固定 checklist，而是**可演进的 evals 体系**：
- 每次踩坑（如某类事实错误反复出现）→ 新增 eval 到审查员检查清单
- 定期复盘失效的 eval（如某条规则已不适用）→ 更新或删除
- Promptfoo 的 CI/CD 防回归思想：改 prompt 时自动跑评估矩阵对比前后，防止「修了 A 味、坏了事实性」

**承载角色**：审查员（[05-reviewer.md](../agents/05-reviewer.md) v2.2）

**现状评估**：本层已成熟，是当前团队最强的一环。v1.1 吸收的方法论是**执行准则升级**（偏见消除 / 原子化 / 工具增强 / 可演进），不新增检查项。按吴恩达观点，本层是基础但不是核心竞争力——继续维护即可，不追加投入。

### 第二层 QA · 读者价值层（PM 型，秘书 + 审查员升级）

**目标**：确保文章「值得读」——从挑错转向以读者价值为核心的需求定义与体验优化。

**触发节点**：Gate 0（需求共识）/ Gate A（初稿确认）/ Gate E（终稿确认）

**核心理念转变**：
- 第一层 QA 问「这篇文章有没有错？」
- 第二层 QA 问「目标读者花 10 分钟读完，能得到什么？这个回报对得起他的时间吗？」

**新增检查项 · 读者价值验证清单**：

| 检查项 | 触发 Gate | 承载角色 | 通过标准 |
|---|---|---|---|
| 读者画像验证 | Gate 0 | 秘书 | 目标读者是谁（具体到人/人群）+ 他花 10 分钟能得到什么（一句话）已写入需求共识 |
| HKRT 价值承诺回扣 | Gate A / Gate E | 审查员 L3 升级 | 文章兑现了 HKRT 评分时承诺的「读者看完应能说的话」（见 TEAM.md §九） |
| 体验曲线验证 | Gate A | 审查员 L3 升级 | 开头 100 字内有钩子；中段无连续 800 字信息密度低谷；结尾有行动号召 |
| 信息密度核查 | Gate A | 审查员 L3 升级 | 每 800 字 ≥1 个新信息点（观点/案例/数据/画面），三段连续无新信息 = 删段 |
| 读者锚点机制 | Gate A | 审查员 L3 升级 | 每 800 字 ≥1 个读者锚点（场景代入 / 利益明示 / 共鸣确认 / 疑问句刹车） |
| 平台适配验证 | Gate E | 秘书 + 排版师 | 文章形态符合目标平台爆款公式（公众号叙事型 / 知乎分析型 / 小红书精简型） |

**审查员 L3 升级说明**：
- 原 L3「内容质量深审（针对目标读者）」以挑错为主——观点支撑、案例密度、逻辑漏洞
- 升级后 L3 增加「读者价值验证」视角——不只问「有没有错」，先问「读者为什么要读完」「读完能得到什么」
- 这是吴恩达说的「QA → PM 转型」：审查员不再只是质检员，而是读者代言人

**秘书在 Gate 0 的升级**：
- 原 Gate 0：7 项必填需求 + HKRT 评级 + 用户确认
- 升级后 Gate 0：增加「读者价值定义」——目标读者具体到人群 + 他花时间读完能得到什么（一句话价值承诺）
- 这句话价值承诺写入工作包「主旨锁」字段，全文对齐

**v1.1 新增方法论吸收**（来自 arxiv 论文调研，不新增检查项，作为读者价值层 QA 执行准则）：

#### 吸收 5：MT-Bench 多轮体验衰减（arxiv: 2306.05685）

**原方法**：单轮评分不可靠，多轮对话中第二/三轮的能力衰减才是真问题。

**写作场景适配**：读者价值不能只看开头钩子（第一轮）。MT-Bench 启示我们做**三段式体验扫描**：
- 第一段（开头 100 字）：钩子是否有效？——读者愿不愿意读下去
- 第二段（中段 800 字 ×N）：信息密度是否持续？——读者会不会读到一半划走
- 第三段（结尾）：读者合上文章能记住什么？——读完是否有"获得感"

**与体验曲线验证的关系**：现有"体验曲线验证"是 MT-Bench 多轮衰减思想的具体落地——中段无连续 800 字低谷即对应"第二轮能力不衰减"。v1.1 把这个对应关系显式化：读者价值层 QA 的本质是**多轮体验衰减检测**，不只看首尾。

#### 吸收 6：Helpful-Honest-Harmless 三维价值框架（arxiv: 2204.05862）

**原方法**：把"好"拆成 Helpful（有用）+ Honest（诚实）+ Harmless（无害），单维高分不够，要全维达标。任一维度 0 分，整体归零。

**写作场景适配**：读者价值 = 三维乘积，任一为 0 总分归零：

| 维度 | 写作场景定义 | 0 分触发条件 |
|---|---|---|
| **Helpful（有用）** | 读者读完有信息增量（新认知/新方法/新案例/新视角） | 全文是常识堆砌 / 信息密度三段连续低于阈值 |
| **Honest（诚实）** | 不夸大不误导，事实可追溯，结论有边界声明 | 关键事实错误 / 标题党与正文不符 / 隐藏关键前提 |
| **Harmless（无害）** | 不焦虑诱导、不制造对立、不消费弱势群体 | 焦虑标题党 / 性别对立挑拨 / 病毒式情绪操纵 |

**硬约束**：Honest 维度与第一层 QA 的"事实核查"重叠——这正常，因为事实性既是"没有错"也是"对读者诚实"。两层 QA 在此交叉，不是冗余而是**双重保险**。

#### 吸收 7：多 Agent 辩论收敛（arxiv: 2305.14325）

**原方法**：多个 LLM 实例就同一问题辩论，收敛到共识，降低单一评判者的偏见。

**写作场景适配**：读者价值不是单人判定。读者价值层 QA 在 Gate A/E 应做**多视角博弈**：
- 审查员（从质检视角）：事实/逻辑/AI 味
- 秘书（从 PM 视角）：读者画像兑现度/价值承诺回扣
- 作者本人（从品味视角）：这个表达是不是我？
- 必要时引入模拟读者 Agent（从读者视角）：我读到第 X 段会不会划走？

**收敛规则**：四视角中任一给出"不通过"，进入修订循环。共识不要求全票通过，但**反对意见必须显式记录**（上下文资产，见 §三）。

**与第一层 LLM-as-judge 偏见消除的关系**：第一层的"异质审查者"是横向（写 vs 查），本层多 Agent 辩论是纵向（同一篇文章不同角色看）。两者互补。

#### 吸收 8：个性化评判标准（Personalized Evaluation）

**原方法**：通用 benchmark 有局限，个性化场景需定制评判标准。读者价值因人而异，没有"通用读者"。

**写作场景适配**：读者价值层 QA 不能用固定阈值一刀切。需在 Gate 0 声明目标读者画像后，**动态调整评判标准**：
- 给专业读者（行业从业者）：信息密度阈值上调，常识不需赘述
- 给大众读者（路人读者）：开头钩子权重上调，专业术语必须翻译
- 给垂直社区（知乎/小红书/公众号）：平台调性决定评判倾向

**与 HKRT 评分的关系**：HKRT 已是"个性化选题评判"的雏形——话题度/知识度/共鸣度/时效度随读者画像加权。v1.1 把这个思想扩展到全文评判：读者画像不仅决定选题是否值得做，还决定初稿/终稿的"值得读"标准如何加权。

**承载角色**：秘书（Gate 0 定义读者画像与价值承诺）+ 审查员 L3 升级（Gate A/E 验证价值兑现）+ 作者本人（品味视角）

**现状评估**：本层是 v1.1 升级的重点。原 v1.0 已有读者价值验证清单，但缺乏方法论支撑——为什么是这些检查项？v1.1 通过 4 个方法论吸收回答：因为读者价值 = 多轮体验不衰减 × 三维全达标 × 多视角共识 × 个性化标准。**按吴恩达观点，本层是核心竞争力**——第一层防错只能保证"不挨骂"，本层才能保证"被转发"。

### 第三层 QA · 长期竞争力层（战略型，风格官 + 作者承载）

**目标**：确保文章「强化作者辨识度 + 积累复利资产」——构建别人无法短期复刻的上下文优势。

**触发节点**：Gate E 后（终稿交付后）+ 季度复盘

**核心理念**：吴恩达说第三层 Loop「最慢但决定产品生死」，其「慢」属性反而构建高壁垒——迭代依赖对用户需求、行业场景的长期深度沉淀，无法通过短期技术投入快速复刻。写作团队的对应物是「作者上下文资产沉淀」。

**新增检查项 · 内容战略自检**：

| 检查项 | 触发时机 | 承载角色 | 通过标准 |
|---|---|---|---|
| 辨识度验证 | Gate E 后 | 风格官 | 这篇文章强化了作者的哪个独特视角？（对比风格官 9 维画像，标注命中的维度） |
| 认知资产积累 | Gate E 后 | 风格官 | 沉淀了什么可复用的认知/方法/案例？（写入风格库对应章节） |
| 差异化验证 | Gate E 后 | 风格官 + 选题官 | 相比同类爆款，这篇文章的「认知剪刀差」在哪？（一句话） |
| 上下文资产沉淀归档 | Gate E 后 | 风格官 | 作者的修改决策是否留痕？（上下文优势显式化，见下节） |
| 长期壁垒评估 | 季度复盘 | 风格官 + 作者 | 近 N 篇是否在构建「别人无法短期复刻」的上下文优势？画像是否在迭代？ |
| 画像命中率 | 季度复盘 | 风格官 | 风格官画像连续 5 篇未被作者修改命中 → 评估画像是否过时（1001 协议） |

**风格官升级说明**：
- 原风格官：画像建立 + 画像注入 + 风格学习沉淀（P0/P1/P2）
- 升级后风格官：增加「上下文资产管理者」职责——不只维护画像，还要评估每篇文章对作者长期辨识度的贡献
- 这是吴恩达说的「品味 = 上下文优势」的落地——品味不可被 AI 替代，本质是人对场景的深度理解

**作者本人的品味判断**（不可委托给 AI）：
- Gate 把关决策（哪些选项被否决及理由）
- 终稿修改 diff + 修改理由
- HKRT 评分时的价值判断
- 选题拍板时的方向选择

这些判断是「品味」的显式化，必须留痕归档，作为作者无法被复刻的核心资产。

**v1.1 新增方法论吸收**（来自 arxiv 论文 + GitHub 工具调研，作为长期竞争力层 QA 执行准则）：

#### 吸收 9：Reflexion 情景记忆（arxiv: 2303.11366）

**原方法**：Agent 失败后生成反思文本，存入记忆缓冲，后续任务复用，避免重复踩坑。

**写作场景适配**：每篇文章交付后，不只是归档上下文资产，还要生成**「本次踩坑反思」**：
- 哪些事实错误是审查员漏检的？为什么？（如：信源单一被误判为可信）
- 哪些 AI 味模式是这次新发现的？（如：某类过渡句反复出现）
- 哪些读者反馈偏差超预期？（如：HKRT 预估高分但实际阅读量低）

**落地形式**：反思文本写入风格库「踩坑日志」章节，下一篇任务启动时审查员先读取最近 N 条反思，作为本次审查的**额外检查项**。这是 v1.1 引入的**可演进 evals**（吸收 4）在长期竞争力层的延伸——不只固化规则，更固化"曾经错过的地方"。

**与现有风格学习机制的关系**：风格学习是 P0/P1/P2 规律提炼（写法层面），Reflexion 是错误反思（事实/逻辑/读者反馈层面）。两者互补：风格学习让 Agent 越来越像作者，Reflexion 让 Agent 越来越不像过去的自己（少犯错）。

#### 吸收 10：Self-Rewarding 自我进化（arxiv: 2401.10020）

**原方法**：模型在生成任务中同时训练评判能力，评判能力随生成能力共同提升，形成正反馈。

**写作场景适配**：写作 Agent 不是被动等待人工反馈，应在每篇文章交付后做**自评**：
- 撰稿人自评：这篇初稿哪里自己不满意？（生成视角的元认知）
- 审查员自评：四层审查哪些判断事后看是错的？（评判视角的元认知）
- 风格官自评：画像注入对最终成文的贡献度几何？（策略视角的元认知）

**硬约束**：自评不替代作者品味判断（吴恩达原话「context advantage」）。自评是 Agent 侧的自我进化，作者侧的品味判断仍是不可替代的最终裁决。两者平行：Agent 越来越准，作者越来越显式化。

#### 吸收 11：风格指纹量化（StyleLLM，GitHub 1k+ Star）

**原方法**：中文文风学习与迁移，把"风格"从主观感受量化为可计算的指纹（句长分布/虚词频率/标点偏好/语义模式）。

**写作场景适配**：风格官的 9 维画像 + Voice Calibration 机械扫描清单（已在 [06-style-profile.md](../agents/06-style-profile.md) v2.0 落地）正是风格指纹量化的实践。v1.1 显式标注这一对应关系，并补充**"指纹漂移检测"**：
- 每篇文章交付后，风格官计算本文指纹与画像基线的偏离度
- 偏离度过大 → 是作者在迭代风格？还是 Agent 生成时偏离了画像？
- 两种情况都需要显式归档（前者是上下文资产，后者是质量问题）

**与画像命中率的关系**：现有"画像命中率"是定性判断（连续 5 篇未命中→评估过时），指纹漂移检测补充了**定量维度**——单篇偏离度即可触发评估，不必等 5 篇。

#### 吸收 12：ExpertPrompting 身份条件生成（arxiv: 2305.14688）

**原方法**：在生成前注入专家身份（"你是 X 领域的资深从业者"），显著提升输出质量。

**写作场景适配**：风格官画像注入已是 ExpertPrompting 的实践——把作者的人设/经历/偏好作为身份条件注入撰稿人。v1.1 把这个思想扩展到**审查与评判阶段**：
- 审查员不应以"通用质检员"身份审查，应以"目标读者画像"身份审查（如：审查一篇给产品经理看的文章时，审查员应代入产品经理视角）
- 这是第二层"个性化评判"（吸收 8）在第三层的延伸——不只评判标准个性化，评判者身份也要个性化

#### 吸收 13：「认知投降」检测（Addy Osmani 三种认知风险）

**原方法**：Addy Osmani 提出 AI 协作的三种认知风险——验证不可省（Verification）/ 理解债务（Comprehension Debt）/ 认知投降（Cognitive Surrender，逐步把判断权让渡给 AI，最终失去独立判断能力）。

**写作场景适配**：长期竞争力层 QA 必须检测作者是否陷入**认知投降**：
- 作者是否越来越快地点"通过"Gate？（Gate 通过时间持续缩短 → 警惕）
- 作者是否越来越少提修改意见？（终稿修改 diff 持续减少 → 警惕）
- 作者是否越来越少做选题拍板？（越来越多"你看着办" → 警惕）

**硬约束**：认知投降是上下文资产的**反向指标**——上下文资产沉淀越多，作者应越来越有判断力，而不是越来越省心。如果季度复盘发现认知投降迹象，必须暂停 Agent 自动化，强制作者深度介入 N 篇做判断校准。

**这是第三层 QA 的"停止条件"**：不只是评估"品味好不好"（上下文资产质量），还要评估"上下文资产沉淀机制是否还在运转"——一旦作者认知投降，整个上下文资产沉淀机制就失效了。呼应概念修正 2：上下文优势的前提是上下文还在持续被显式化，AI 全权代理 = 上下文优势流失。

**承载角色**：风格官（Gate E 后评估 + 季度复盘）+ 作者本人（品味判断显式化 + 认知投降自检）

**现状评估**：本层是 v1.0 的薄弱环节——v1.0 只有定性检查项，缺乏方法论支撑。v1.1 通过 5 个方法论吸收回答了"长期竞争力如何被度量"：Reflexion（错误不重复）+ Self-Rewarding（Agent 自进化）+ 风格指纹量化（漂移可检测）+ ExpertPrompting（身份条件贯穿）+ 认知投降检测（品味机制保活）。**按吴恩达观点，本层决定团队生死**——前两层可以被短期投入复刻，本层的上下文资产只能靠时间沉淀。

## 三、上下文资产沉淀机制（贯穿三层，把已有资产显式化）

**核心理念**（v1.1 更新，呼应概念修正 2）：吴恩达用「上下文优势」替换「品味」概念，因为品味暗示不可传授，上下文优势指向可操作路径——把独有上下文显性化，AI 就能变好。本节把分散在各角色卡中的「上下文优势」显式归档为「上下文资产」，让上下文优势可积累、可回看、可迭代、可被 AI 复用。

> 命名说明：v1.0 称「品味资产」，v1.1 统一改为「上下文资产」，与吴恩达原文术语对齐。下文涉及「品味」处均指「上下文优势的显式化」。

### 上下文资产清单（每篇交付后归档）

| 资产 | 来源角色 | 归档位置 | 价值 | v1.1 新增 |
|---|---|---|---|---|
| 作者 Gate 把关决策 | 作者 + 秘书 | 工作包「决定」字段 | 哪些选项被否决及理由——上下文优势的方向感 | |
| 作者终稿修改 diff + 理由 | 作者 + 审查员 | 风格学习移交包（P0/P1/P2） | 为什么改——上下文优势的颗粒度 | |
| 风格官 Voice Calibration | 风格官 | 风格库（9 维画像 + 机械扫描清单） | 句式节奏/用词/怪癖的量化沉淀 | |
| 选题官认知剪刀差 | 选题官 | 选题卡片 | 从同类爆款提炼的差异化角度 | |
| HKRT 价值承诺 | 选题官 + 秘书 | 工作包「主旨锁」字段 | 话题度/知识度/共鸣度/时效度的读者校准 | |
| 素材契约主旨锁 | 研究员 + 撰稿人 | 草稿头部 | 作者一句话复述核心主旨与情绪基调 | |
| 读者实际反馈偏差（如有） | 排版师 + 作者 | 季度复盘 | HKRT 评分与实际读者反馈的偏差——上下文校准 | |
| **踩坑反思日志** | 审查员 + 撰稿人 | 风格库「踩坑日志」章节 | 本次漏检的事实错误/新发现 AI 味模式/读者反馈偏差——避免重复踩坑（吸收 9 Reflexion） | ✅ |
| **Agent 自评记录** | 撰稿人/审查员/风格官 | 工作包「自评」字段 | 三个角色各自的自评——Agent 侧自我进化（吸收 10 Self-Rewarding） | ✅ |
| **风格指纹漂移度** | 风格官 | 风格库「指纹漂移」章节 | 本文指纹与画像基线的偏离度——区分作者迭代 vs Agent 偏离（吸收 11 StyleLLM） | ✅ |
| **认知投降指标** | 秘书（自动统计） | 季度复盘 | Gate 通过时间趋势/终稿修改 diff 数量趋势/选题拍板介入率——品味机制保活（吸收 13） | ✅ |

### 上下文资产沉淀流程（v1.1 扩展）

1. **Gate 把关时**：秘书记录作者决策（选项 + 理由），写入工作包「决定」字段
2. **终稿修改时**：审查员逐段对比 AI 原稿与作者修改版，提炼 P0/P1/P2 规律 + 修改理由
3. **Gate E 后**：风格官接收移交包，评估写入风格库（写入前快照备份）
4. **Gate E 后·踩坑反思**（v1.1 新增）：审查员 + 撰稿人生成「本次踩坑反思」，写入风格库「踩坑日志」章节
5. **Gate E 后·Agent 自评**（v1.1 新增）：三个角色各自生成自评记录，写入工作包「自评」字段
6. **Gate E 后·指纹漂移检测**（v1.1 新增）：风格官计算本文指纹与画像基线偏离度，写入风格库「指纹漂移」章节
7. **季度复盘**：风格官汇总近 N 篇上下文资产，评估画像迭代方向 + 长期壁垒强度 + 认知投降指标

**硬约束**（v1.1 扩展）：
- 上下文资产归档必须含「理由」，不只记「改了什么」还要记「为什么改」。无理由的修改记录 = 不可写入上下文资产。
- 踩坑反思必须含「根因」+「下次检查项」，无根因的反思 = 不可写入踩坑日志。
- 自评必须含「不满意处」+「判断事后看是否错的维度」，全满意的自评 = 不可写入自评记录（Self-Rewarding 的自评应识别不足，不是自我表扬）。
- 认知投降指标任一触发阈值 → 季度复盘必须强制作者深度介入下 N 篇。

## 五、角色融合说明（吴恩达「角色融合与边界重构」落地，v1.1 扩展）

吴恩达洞察：在三层 Loop 推动下，程序员、QA、产品经理的角色边界正在被打破，走向能力交叉融合。映射到写作团队：

| 角色 | 原定位 | 能力边界延伸 | 对应 Loop 层 | v1.1 新增承载 |
|---|---|---|---|---|
| 审查员 | 质检官（挑错） | QA → PM：L3 从挑错升级为读者价值验证，做读者代言人 | 第二层 | 吸收 1 偏见消除 / 吸收 2 FActScore / 吸收 3 CRITIC 双路审查 / 吸收 9 踩坑反思 / 吸收 10 自评 / 吸收 12 评判身份代入 |
| 秘书 | 调度官（编排） | 调度 → PM：Gate 0 做读者价值定义，不只接活还要定义价值 | 第二层 | 吸收 8 个性化评判标准动态调整 / 吸收 13 认知投降指标自动统计 |
| 风格官 | 画像维护 | 画像维护 → 上下文资产管理：Gate E 后做长期竞争力评估 | 第三层 | 吸收 11 风格指纹漂移检测 / 吸收 10 画像贡献度自评 / 吸收 12 画像注入即 ExpertPrompting |
| 作者本人 | 把关人 | 上下文优势显式化留痕——不可被 AI 替代的核心资产 | 第三层 | 吸收 13 认知投降自检（品味机制保活的最终裁决者） |

**不新增角色**：按吴恩达洞察，QA → PM → 战略是能力升级不是组织扩张。新增角色会打破现有 8 角色单角色单标准的清晰边界。v1.1 的 13 个方法论吸收全部由现有 4 个角色承载（审查员 / 秘书 / 风格官 / 作者），不增加组织复杂度。

**审查员与秘书的边界**（v1.1 澄清）：
- 审查员 L3 升级是「执行层读者价值验证」（针对单篇文章，吸收 5/6/7）
- 秘书 Gate 0 升级是「需求层读者价值定义」（针对任务本身，吸收 8）
- 两者互补：秘书定义价值，审查员验证价值是否兑现
- **v1.1 新增边界**：秘书还承载「认知投降指标自动统计」（吸收 13），审查员承载「踩坑反思 + 自评」（吸收 9/10）——这是数据采集 vs 内容反思的分工，不重叠

**审查员与风格官的边界**（v1.1 新增）：
- 审查员在 Gate E 后做「踩坑反思」（错误层面，写入踩坑日志）
- 风格官在 Gate E 后做「指纹漂移检测」（风格层面，写入指纹漂移章节）
- 两者互补：审查员关注"对不对"，风格官关注"像不像"

## 六、能力提升总结（v1.1 扩展）

本 QA 框架对 Agent Team 的能力提升体现在六个维度：

1. **从挑错型 QA 升级为价值定义型 QA**（第二层）：审查员不再只问「有没有错」，先问「读者为什么要读完」——吸收 5/6/7/8 提供方法论支撑
2. **从单篇交付升级为长期上下文资产沉淀**（第三层）：每篇文章都在积累作者辨识度 + 认知资产 + 踩坑反思 + 自评记录，构建别人无法短期复刻的壁垒——吸收 9/10/11/12/13 把定性检查升级为可度量机制
3. **显式化已有的上下文资产**：把分散在各角色卡中的「上下文优势」归档为可积累、可回看的资产清单（v1.1 从 7 项扩展到 11 项）
4. **角色边界融合**：QA 不再是审查员一个人的事，秘书（PM 视角 + 认知投降监控）、风格官（战略视角 + 指纹漂移）、作者（上下文优势显式化 + 认知投降自检）共同承载
5. **执行准则升级**（v1.1 新增）：第一层 QA 通过 4 个方法论吸收（偏见消除 / 原子化 / 工具增强 / 可演进）从 checklist 升级为方法论驱动的执行准则
6. **停止条件机制**（v1.1 新增）：第三层 QA 引入「认知投降检测」——不只评估"上下文资产好不好"，还评估"上下文资产机制是否还在运转"。一旦作者认知投降，强制深度介入做校准

**与现有去 AI 味策略的关系**：
- 去 AI 味（TEAM.md §六）是第一层 QA 的核心，已完善
- 本 QA 框架是在去 AI 味基础上叠加第二层（读者价值）和第三层（长期竞争力）
- 三层叠加：文章既「没有 AI 味」（第一层），又「值得读者花时间」（第二层），还在「强化作者辨识度」（第三层）
- v1.1 新增：第三层还保证「作者判断力不退化」（认知投降检测）——这是去 AI 味战略的长期保障，AI 味的根源之一就是作者认知投降后 Agent 全权代理

## 七、接入说明（v1.1 扩展）

1. **本框架不替代审查员**：第一层 QA 由审查员继续承载，第二/三层是能力延伸
2. **不新增 Gate**：三层 QA 触发节点都映射到现有 6 个 Gate + Gate E 后 + 季度复盘
3. **最小改动落地**：审查员 L3 升级 + 秘书 Gate 0 升级 + 风格官 Gate E 后升级，均可通过角色卡补充实现
4. **上下文资产归档**（v1.1 扩展）：每篇交付后由风格官汇总上下文资产清单（11 项），写入风格库对应章节；审查员同步生成踩坑反思日志；三角色各自生成自评记录
5. **季度复盘**（v1.1 扩展）：建议每季度由风格官 + 作者做一次长期壁垒评估 + 画像命中率核查 + 认知投降指标核查
6. **认知投降阈值**（v1.1 新增）：建议初始阈值——Gate 通过时间连续 3 篇缩短 ≥20% / 终稿修改 diff 连续 3 篇减少 ≥30% / 选题拍板介入率连续 3 篇为 0。任一触发即暂停 Agent 自动化，强制作者深度介入下 2-3 篇
7. **可演进 evals**（v1.1 新增，吸收 4）：本框架不是固定 checklist，每次踩坑应新增 eval 到审查员检查清单；定期复盘失效的 eval 应更新或删除。Promptfoo 的 CI/CD 防回归思想可参考——改 prompt 时跑评估矩阵对比前后
