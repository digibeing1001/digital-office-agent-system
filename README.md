# Digital Office 数字办公室

> 一句话定位：你只管把事情说清楚，秘书负责找人、盯进度、管审批；研究、产品、设计、开发、写作、规划、法务，各有一位对结果负责的数字员工。

## 这是什么

Digital Office 是一套能装在你电脑里的「数字办公室」。它不是一个会自己乱跑的 AI 助手，而是一家**有人负责、交代得清楚、不会瞎循环、关键步骤有人把关**的数字公司。

打个比方：市面上的 AI 助手像一个什么都能干一点的临时工，你交代一句它做一件，下一件就不记得上一件了；而 Digital Office 更像一家真实的小公司——你只跟秘书说一件事，秘书会先确认你到底要什么，再把活儿交给最合适的数字员工去做，做完把结果交回给你。整个过程你都看得见、管得住，任务走到哪一步、上一步交了什么、为什么停住，全都查得到。

它不追求「比别人更聪明」，而是让你**随时知道事情现在在哪里**。

---

## 它能帮你做什么

下面这些场景，你只要跟秘书说一句话，剩下的交给数字员工：

- **做调研**：研究同类产品、查市场数据、整理带出处的证据，给你一份能直接拿去开会的研究报告。
- **写需求文档**：把「我想做一个 XX」这种模糊想法，变成清晰的产品需求文档，列好优先级和验收标准。
- **排项目计划**：把一个大目标拆成几步、定好里程碑、标出依赖和风险，让你知道先做什么后做什么。
- **设计界面**：从用户流程到视觉稿，画好原型并做无障碍检查，给程序员一份能直接照着实现的稿子。
- **写代码**：实现新功能、修 Bug、做测试、跑构建验证，交付前过质检门禁，不是「能跑就行」。
- **写文章文案**：把零散材料整理成文章、产品文案、发布说明、用户文档，做事实核对和润色。
- **审合同查合规**：审查供应商合同、隐私条款、产品合规风险，给出企业内部审查草稿（最终外发仍需人类律师把关）。
- **串联多步**：一句「先调研同类产品，再整理需求，设计界面，最后实现一个能跑的前端版本」，秘书会自动安排研究→产品→设计→开发这条流水线。

## 七位数字员工

你不用记每个员工叫什么，秘书会自动判断该找谁。但你最好知道这家公司一共有这几位：

| 数字员工 | 负责什么 | 主要用到的本事 |
| --- | --- | --- |
| 秘书 | 接活、确认你到底要什么、分派、盯进度、管审批 | 团队排班、找技能、资料运营 |
| 研究员 | 查资料、找证据、写引用、做调研 | 迭代检索、深度研究、市场调研 |
| 产品经理 | 把模糊需求变成清晰任务、排优先级、定验收 | 需求澄清、需求文档起草 |
| 规划师 | 拆解项目、排时间线、标依赖和风险 | 头脑风暴、写计划 |
| 设计师 | 界面流程、视觉、原型、无障碍 | 设计、设计质检 |
| 程序员 | 写代码、调试、测试、部署准备 | 编码质检、测试驱动、编码规范 |
| 写作员 | 提纲、起草、编辑、终稿 | 写作、文案、内容策略 |
| 数字律师 | 合同审查、隐私数据、产品合规、用工与知识产权、争议分流 | 法务工作流（按通道干活） |

每位数字员工对一个方向负责。**不会出现三四个 AI 抢同一件事**，也不会层层上报走流程。数字员工内部用到的「技能」只是工具，不是另一个人。

## 八大特点

和一般 AI 助手比，这套系统好在八件事上，每件都和「再多装一个 AI」的做法相反。

### 1. 一个秘书入口，七位数字员工

你只需要对秘书说一件事，秘书负责判断该找谁、要不要先问清楚、哪些步骤需要审批，然后把活儿交给一位明确的数字员工。**每件事只有一个负责人**，不会出现「三四个 AI 抢同一件事」的混乱。

### 2. 四步干活法，不会无限循环

每位数字员工干活都按同样的四步走，每一步都记下来、可以回看：

1. **搞清楚**：拿到下一步需要的可靠信息
2. **定路线**：形成决定、方向、风险和下一步动作
3. **去执行**：调用数字员工和工具把活儿干出来
4. **看结果**：对照证据、验收标准、进度和预算判断成不成

关键区别：**后台控制系统说了算，AI 只能建议**。AI 可以提下一步，但只有后台控制系统能在 8 种状态间切换（继续 / 改计划 / 重试 / 等人 / 完成 / 失败 / 取消 / 预算用尽）。循环次数、重试、时长、工具调用、模型调用、花费都有上限，进度停滞会被发现——**任务不会无限自我反思下去、烧你的钱**。

### 3. 任务交接不靠聊天记忆，靠带版本的「工作包」

这是这套系统最值钱的地方。任务从一个数字员工转给下一个时，系统不会把整段聊天记录一股脑塞过去，而是生成一个**紧凑、带版本、可校验**的工作包，里面写清楚：

- 你真正想做成什么、当前目标、限制和验收标准
- 哪些是确定的事实、哪些是假设、哪些还不确定或有冲突，各自有多可信
- 证据和产物放在哪里（按引用取回，不重复传大文件）
- 已经做过哪些决定、哪些内容被省略了以及怎么找回
- 交给谁了、对方有没有核对过交接印记、有没有要求补充信息

接收方必须明确点「收到」（可以「收到 / 要补充信息 / 退回」），发送方不能算自己干完了。**AI 的内心独白、密码、密钥、登录令牌一概不许进工作包**。

### 4. 关键动作有人把关，AI 不能自己拍板

不是什么动作都能让 AI 自己决定。系统列出 **11 类硬管控动作**，必须经过人工判断或专业复核才能放行：

> 对外发送、公开发布、上线部署、删除、覆盖、付款、数据导出、记入长期记忆、晋升为公司知识、给出受监管的专业建议、最终交付。

系统按 5 类风险（受监管的专业领域、不可逆或对外动作、知识晋升、权限范围不清、证据缺失或冲突）自动评估风险分，超过门槛（默认 0.65）就暂停并请人来确认。**AI 可以请求暂停，但只有后台控制系统能冻结和恢复**——这避免了 AI 绕过你自行其是。

### 5. 本地 AI 和在线服务，你想用哪个用哪个

你可以同时用本地装的 AI 工具和在线大模型服务，系统会按你定的策略自动选路：

- **本地 AI 工具**：自动发现 Hermes、OpenClaw、Codex、Claude Code
- **在线服务**：MiniMax、MiMo、Kimi、智谱 GLM、OpenAI、Anthropic、Gemini，或任意 OpenAI 兼容服务
- **套餐服务**：MiniMax Coding Plan、MiMo Token Plan、Kimi Code，或你自己填的兼容套餐

每个连接可以改地址、改模型、改密钥，还能做一次真实的小请求测试。密钥存在服务器本机一个只有你能读的文件里，浏览器、界面、审计和日志只能看到打码后的尾号。**每次选了哪条路、调用了多少、AI 产出了什么、有没有失败，全部写进记录——没有可用连接时会明确告诉你失败了，不会假装成功**。

### 6. 资料分三个边界，谁能看什么清清楚楚

资料库不是一个乱糟糟的大池子，而是三类边界清晰的空间：

- **公司资料**：共享规则、事实、已批准的方法，全员可见
- **项目资料**：项目专属的文档、资产、决定，只有这个项目的成员能看
- **授权行业资料**：外部购买的行业知识，按授权范围挂载，不复制到本地

权限分「只读 / 可写 / 可管」三级。当项目事实和公司知识冲突时，**项目源文件在「这个项目的实际情况」层面优先，公司知识在「全公司统一策略」层面优先，安全规则始终最高**。数字员工用资料库里的内容时必须标明出处。

### 7. 发布前必须过质检，不是「跑通了就行」

每次发版前必须通过 5 个角色质检（产品 / 设计 / 实现 / 写作 / 交付），每个质检都有入口要求、出口产物和确定性检查：

- 写代码这条强制：构建通过、确定性测试通过、冒烟测试通过、密钥扫描没发现泄漏、改动可审阅、没有偷偷扩大范围
- 原则明确：**证据优先，确定性检查为主，AI 审查为辅，没批准不直接上生产**
- 所有产物可记录、可回滚

### 8. 企业数字律师，不是万能律师提示词

法务统一为一位**企业内部数字律师**，按 6 条通道干活：法务收口、合同审查、隐私数据、产品合规、用工与知识产权、争议分流。不是那种「你是个律师」一句话提示词就开干的万能律师。

所有法律输出都是**企业内部审查草稿**，不替代执业律师意见。正式外发、签署、用工动作、产品上线、诉讼或仲裁必须经过合适的人类专业复核。

## 与其他分支的关系

这个仓库有三个并列分支，各自服务不同场景，但共享同一套工程底座：

| 分支 | 定位 | 适合谁 |
| --- | --- | --- |
| **main**（你现在看的这个） | 数字办公系统 | 通用企业办公，七角色齐全，覆盖产品/设计/研发/法务/写作全链路 |
| **research-team** | 科研团队特化 | 做科研的人，强调研究完整性门控、评分轨迹追踪、状态可观测 |
| **writer-team** | 写作团队特化 | 做内容的人，8 角色写作流水线 + 去 AI 味、读者价值验证 |

三个分支不互相替代，而是各管一摊。如果你做的是通用办公，留在 main；如果是纯科研，看 research-team；如果是纯内容创作，看 writer-team。

## 怎么用

### 三步上手

**第一步：一条命令装好**

在 WSL 或 Linux 终端运行：

```bash
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/install.sh | bash
```

它会装到 `~/.hermes`，装好技能，并自动跑健康检查和发布质检。以后升级只需要：

```bash
~/.hermes/update
```

升级**只更新软件、配置和内置技能**，不会把你电脑里的登录缓存、会话、项目、资料库、任务、审批或个人偏好搬走，也**不会在升级时覆盖这些数据**。

**第二步：启动界面**

```bash
# 普通用户界面
digital-office-gui

# 或者用后台命令
office-system

# 管理后台
digital-office-gui --admin

# 后台常驻运行
digital-office-gui --background
```

系统优先用 `http://127.0.0.1:8787/`，端口被占了就自动找下一个可用的。浏览器可以把这个界面装成桌面应用。**如果你要让别的电脑访问**，必须配一个访问令牌（`DIGITAL_OFFICE_WEB_TOKEN` 或 `--token`），不会把项目和任务状态裸露在局域网上。

**第三步：跟秘书说话**

直接对秘书说一件事就行，例如：

```text
帮我审查这份供应商合同，列出高风险条款和需要业务确认的问题。
```

```text
先调研同类产品，再整理需求，设计界面，最后实现一个能跑的前端版本。
```

```text
把这份材料写成一篇文章，完成事实核对、编辑和终稿检查。
```

### 一个提醒：新项目不会「一句话就开跑」

你说完第一句话，秘书不会立刻动手。他会先复述他对你意图的理解，请你确认，然后至少问你三个层层深入的问题，让你说清楚：真正要改变的结果、谁来判断成功、交付物、不能妥协的边界、最大能接受的失败、原始资料在哪里。**项目准备度达到门槛、你确认了底稿之后，正式工作才会启动。如果之后核心目标变了，原来的确认会自动失效**，要重新确认。

### 常用命令

| 命令 | 作用 |
| --- | --- |
| `digital-office-gui` | 启动普通用户界面 |
| `digital-office-gui --admin` | 启动管理后台 |
| `office-system` | 启动后台服务 |
| `office-system health` | 查看系统健康状态 |
| `agent-router --health` | 查看数字员工路由健康 |
| `~/.hermes/update` | 升级到最新版 |

### 如果你电脑里已经有 Hermes 或个人规则

如果目标目录里已经有不是 Digital Office 管理的规则或个人数据，安装器会停下来让你明确选——**这是刻意的保护，不会替你偷偷决定**：

```bash
# 保留原来的规则，Digital Office 并排安装
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/install.sh | bash -s -- --preserve-existing

# 备份原规则后，让 Digital Office 成为默认秘书入口
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/install.sh | bash -s -- --overwrite-existing

# 装到 OpenClaw
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/install.sh | bash -s -- --host openclaw
```

## 质量保障

这套系统不是「跑通了就交付」，而是分三层把关。通俗说就是：

- **第一层·执行自测**（快）：代码质量、迭代有没有空转、副作用会不会重复出问题、回归有没有被守住。这一层最成熟，但不追求完美——**有记录的小缺陷可以进下一阶段**，不为代码优雅过度迭代。
- **第二层·产品品味**（中速）：需求有没有洞察、体验有没有覆盖完整用户旅程、场景有没有漏掉边界和失败情况。产品经理在需求阶段就预定义验收标准，不只事后验收，**有权叫停「功能正确但价值不足」的迭代**。
- **第三层·产品生死**（慢）：产品方向有没有偏离核心价值、品牌一致性有没有守住、长期健康有没有恶化（技术债、知识库质量、用户满意度趋势）。这一层**人类主导，跨任务累积评估**，不在单次任务里触发。

除了三层质检，还有前面提到的 **11 类关键动作必须人工把关**：对外发送、公开发布、上线部署、删除、覆盖、付款、数据导出、记入长期记忆、晋升为公司知识、给出受监管的专业建议、最终交付。这些动作 AI 不能自己拍板，超过风险门槛就暂停请人确认。

2026-07 还吸收了两项来自其他分支的失效模式防御：

- **反 AI 模板垃圾检查**：防止界面产出千篇一律的「AI 味」模板（紫蓝渐变 Hero、深色网格背景、椭圆按钮堆叠卡片这类），设计师在交付前必须过一遍。
- **认知投降防御**：人类在审批时必须真读改动、写一段不少于 30 字的理解总结才能放行，防止「看都不看就点同意」。

## 不只是这七位员工

上面这七位是系统出厂时配好的「默认团队」。但 Digital Office 本质上是一个**能自己编排任意数字员工团队**的框架，不绑死任何岗位：

- **加人**：管理员可以在界面里创建新的数字员工，起名字、定职责、配技能、绑模型，立刻就能被秘书调度
- **换团队**：如果你的业务用不上法务或设计，只要研究和写作，可以整套换掉团队配置
- **自主编排**：秘书会按你定义的员工清单和各自能力，自动判断该找谁、要不要并行干、要不要等人补充信息
- **自带约束**：不管团队怎么换，「有人负责、交代清楚、不瞎循环、关键步骤有人把关」这套机制都跟着走

换句话说，**这套机制装在什么团队上都成立**。当前这七位只是给你上手用的一个完整示例。

## 两个界面，一套后台

界面分两个入口，但背后是同一套受控后端，业务状态不会各存一份互相打架：

- **`/` 我的办公室**（你日常用）：左侧是「我的办公室、数字员工、资料库」，底部是「项目文件夹、设置」。以项目为核心——你跟秘书说一件事，秘书引导你新建或选项目；每个项目有自己的资料、对话、审批、交付物和工作记录。
- **`/admin` 管理中心**（管理员用）：系统概览、数字员工管理、技能、运行状态、模型接入、策略与预算、审计、系统维护。

界面只读状态、不能自己改工作流——**所有会改变任务、审批或数字员工的操作都走专用后端接口，界面不能自己伪造状态，也不能执行任意系统命令**。

当前版本 `0.3.0 internal`，第一版可用界面已上线：

- 数字员工和技能的责任模型已统一，法务已对齐为企业内部数字律师
- 四步干活法、带版本工作包交接、权限、审批、恢复和审计都已落地
- 完整发布质检和冒烟测试已通过，Hermes / OpenClaw 安装升级路径已具备
- 用户端和管理后台可正常构建和运行，网页可装成桌面应用，接口有认证边界
- 关键写操作有权限、确认和审计，数字员工生命周期已纳入发布质检和网页冒烟测试

也就是说，**现在可以用于本机办公、产品演示和内部试用**；进入企业稳定发布前，仍需经过试点部署、真实业务数据验证和稳定版发布流程。
