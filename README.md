# Digital Office 数字办公室

> 你只管把事情说清楚，秘书负责找人、盯进度、管审批；研究、产品、设计、开发、写作、规划、法务，各有一位对结果负责的数字员工。

Digital Office 是一套可以装在你电脑里的「数字办公室」。它不是一个会自己乱跑的 AI 助手，而是一家**有人负责、交代得清楚、不会瞎循环、关键步骤有人把关**的数字公司。

你用它，就像用一家真公司：跟秘书说一件事，秘书会先确认你到底要什么，再把活儿交给最合适的数字员工去做，做完把结果交回给你。整个过程你都看得见、管得住。

---

## 一分钟看懂

把它想象成一家能跑真实业务的数字公司，但它刻意不模拟层层汇报的人类组织：

```text
你
└── 秘书：接活、确认你要什么、分派、盯进度、管审批
    ├── 研究员 ── 检索、找证据、写引用
    ├── 产品经理 ── 梳理需求、排优先级、定验收
    ├── 设计师 ── 交互、视觉、原型、无障碍
    ├── 程序员 ── 写代码、调试、测试、部署
    ├── 写作 ── 提纲、起草、编辑、终稿
    ├── 规划师 ── 架构、里程碑、依赖、风险
    └── 数字律师 ── 合同、隐私、合规、用工/知识产权
```

**每位数字员工对一件事负责，他内部用到的「技能」只是工具。** 不会出现三四个 AI 抢同一件事，也不会层层上报走流程。

## 它到底好在哪

很多人用过多 AI 工具，都踩过同一个坑：**任务一转手，目标、证据、限制和已经做过的决定就开始丢**。AI 越多，反而越不知道事情走到哪了。

Digital Office 的好，不是它比别人更聪明，而是它让你**随时知道事情现在在哪里、为什么停住、上一步交了什么、出了问题怎么恢复**。具体体现在四件事上，每件都和「再多装一个 AI」的做法相反：

| 别人怎么做 | Digital Office 怎么做 |
| --- | --- |
| 每个小步骤都再开一个 AI | 一位数字员工管一个方向，小步骤只是他内部用的工具 |
| 把整段聊天记录转给下一个 AI | 只传真正需要的部分，大文件按引用取回 |
| 让 AI 自己决定要不要继续做下去 | 后台控制系统决定继续、重做、等人、完成还是放弃 |
| 做完没做完主要看 AI 说「完成了」 | 必须满足验收条件、有产物、过质检、留可回看记录 |
| 更新软件时顺便覆盖你的数据 | 软件文件和你的项目、资料、偏好、密码严格分开 |
| 法务 = 一个万能律师提示词 | 一位企业内部数字律师，分合同/隐私/合规等通道干活，关键动作留人工复核 |

## 八个核心特点

### 1. 一个秘书入口，七位数字员工

你只需要对秘书说一件事，秘书负责判断该找谁、要不要先问清楚、哪些步骤需要审批，然后把活儿交给一位明确的数字员工。**每件事只有一个负责人**，不会出现「三四个 AI 抢同一件事」的混乱。

### 2. 四步干活法，不会无限循环

每位数字员工干活时，都按同样的四步走，每一步都被记录下来、可以回看：

1. **搞清楚**：拿到做下一步需要的可靠信息
2. **定路线**：形成决定、方向、风险和下一步动作
3. **去执行**：调用数字员工和工具把活儿干出来
4. **看结果**：对照证据、验收标准、进度和预算判断成不成

**关键区别：后台控制系统说了算，AI 只能建议。** AI 可以提出下一步，但只有后台控制系统能在 8 种明确状态间切换：继续 / 改计划 / 重试 / 等人 / 完成 / 失败 / 取消 / 预算用尽。

每次干活都有预算护栏——循环次数、重试、时长、工具调用、模型调用、花费都有上限，进度停滞会被发现，**任务不会无限自我反思下去、烧你的钱**。

### 3. 任务交接不靠聊天记忆，靠带版本的「工作包」

这是 Digital Office 最值钱的地方。任务从一个数字员工转给下一个时，系统不会把整段聊天记录一股脑塞过去，而是生成一个**紧凑、带版本、可校验**的工作包，里面写清楚：

- 你真正想做成什么、当前目标、限制和验收标准
- 哪些是确定的事实、哪些是假设、哪些还不确定或有冲突，各自有多可信
- 证据和产物放在哪里（按引用取回，不重复传大文件）
- 已经做过哪些决定、哪些内容被省略了以及怎么找回
- 交给谁了、对方有没有核对过交接印记、有没有要求补充信息

接收方必须明确点「收到」（可以「收到 / 要补充信息 / 退回」），发送方不能算自己干完了。**AI 的内心独白、密码、密钥、登录令牌一概不许进工作包**。

### 4. 关键动作有人把关，AI 不能自己拍板

不是什么动作都能让 AI 自己决定。系统列出 **11 类硬管控动作**，必须经过人工判断或专业复核才能放行：

**对外发送、公开发布、上线部署、删除、覆盖、付款、数据导出、记入长期记忆、晋升为公司知识、给出受监管的专业建议、最终交付**。

系统按 5 类风险（受监管的专业领域、不可逆或对外动作、知识晋升、权限范围不清、证据缺失或冲突）自动评估风险分，超过门槛（默认 0.65）就暂停并请人来确认。**AI 可以请求暂停，但只有后台控制系统能冻结和恢复**——这避免了 AI 绕过你自行其是。

### 5. 本地 AI 和在线服务，你想用哪个用哪个

你可以同时用本地装的 AI 工具和在线大模型服务，系统会按你定的策略自动选路：

- **本地 AI 工具**：自动发现 Hermes、OpenClaw、Codex、Claude Code
- **在线服务**：MiniMax、MiMo、Kimi、智谱 GLM、OpenAI、Anthropic、Gemini，或任意 OpenAI 兼容服务
- **套餐服务**：MiniMax Coding Plan、MiMo Token Plan、Kimi Code，或你自己填的兼容套餐

每个连接可以改地址、改模型、改密钥，还能做一次真实的小请求测试。密钥存在服务器本机的一个只有你能读的文件里（权限 0600），浏览器、界面、审计和日志只能看到打码后的尾号。**每次选了哪条路、调用了多少、AI 产出了什么、有没有失败，全部写进记录——没有可用连接时会明确告诉你失败了，不会假装成功**。

### 6. 资料分三个边界，谁能看什么清清楚楚

资料库不是一个乱七八糟的大池子，而是三类边界清晰的空间：

- **公司资料**：共享规则、事实、已批准的方法，全员可见
- **项目资料**：项目专属的文档、资产、决定，只有这个项目的成员能看
- **授权行业资料**：外部购买的行业知识，按授权范围挂载，不复制到本地

权限分「只读 / 可写 / 可管」三级，可以授予用户、角色、数字员工、项目或工作流。当项目事实和公司知识冲突时，**项目源文件在「这个项目的实际情况」层面优先，公司知识在「全公司统一策略」层面优先，安全规则始终最高**。数字员工用资料库里的内容时必须标明出处。

### 7. 发布前必须过质检，不是「跑通了就行」

每次发版前必须通过 5 个角色质检（产品 / 设计 / 实现 / 写作 / 交付），每个质检都有入口要求、出口产物和确定性检查：

- 写代码这条强制：构建通过、确定性测试通过、冒烟测试通过、密钥扫描没发现泄漏、改动可审阅、没有偷偷扩大范围
- 原则明确：**证据优先，确定性检查为主，AI 审查为辅，没批准不直接上生产**
- 所有产物可记录、可回滚

### 8. 企业数字律师，不是万能律师提示词

法务统一为一位**企业内部数字律师**，按 6 条通道干活：法务收口、合同审查、隐私数据、产品合规、用工与知识产权、争议分流。

所有法律输出都是**企业内部审查草稿**，不替代执业律师意见。正式外发、签署、用工动作、产品上线、诉讼或仲裁必须经过合适的人类专业复核。本地已安装 Apache-2.0 的中文法律技能包；另一份 CC BY-NC-ND 4.0 的资料因许可限制只登记状态，不会启用。

## 三步用起来

### 第一步：一条命令装好

在 WSL 或 Linux 终端运行：

```bash
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/update | bash
```

它会装到 `~/.hermes`，装好技能，并自动跑健康检查和发布质检。以后升级只需要：

```bash
~/.hermes/update
```

升级**只更新软件、配置和内置技能**，不会把你电脑里的登录缓存、会话、项目、资料库、任务、审批或个人偏好搬走，也**不会在升级时覆盖这些数据**。

### 第二步：启动界面

```bash
# 普通用户界面
~/.hermes/digital-office-gui

# 管理后台
~/.hermes/digital-office-gui --admin

# 后台常驻运行
~/.hermes/digital-office-gui --background
```

系统优先用 `http://127.0.0.1:8787/`，端口被占了就自动找下一个可用的。浏览器可以把这个界面装成桌面应用。**如果你要让别的电脑访问**，必须配一个访问令牌（`DIGITAL_OFFICE_WEB_TOKEN` 或 `--token`），不会把项目和任务状态裸露在局域网上。

### 第三步：跟秘书说话

直接对秘书说一件事就行，例如：

```text
帮我审查这份供应商合同，列出高风险条款和需要业务确认的问题。
```

```text
先调研同类产品，再整理需求，设计界面，最后实现一个能跑的前端版本。
```

```text
把这份材料写成一篇文章，完成事实核对、编辑和终稿检查。
```

### 新项目不会「一句话就开跑」

你说完第一句话，秘书不会立刻动手。他会先复述他对你意图的理解，请你确认，然后至少问你三个层层深入的问题，让你说清楚：真正要改变的结果、谁来判断成功、交付物、不能妥协的边界、最大能接受的失败、原始资料在哪里。**项目准备度达到门槛、你确认了底稿之后，正式工作才会启动。如果之后核心目标变了，原来的确认会自动失效**，要重新确认。

## 两个界面，一套后台

界面分两个入口，但背后是同一套受控后端，业务状态不会各存一份互相打架：

- **`/` 我的办公室**（你日常用）：左侧是「我的办公室、数字员工、资料库」，底部是「项目文件夹、设置」。以项目为核心——你跟秘书说一件事，秘书引导你新建或选项目；每个项目有自己的资料、对话、审批、交付物和工作记录
- **`/admin` 管理中心**（管理员用）：系统概览、数字员工管理、技能、运行状态、模型接入、策略与预算、审计、系统维护

你这边把秘书对话放在主工作区中央，看板分区可以拖动排序、切换宽度，布局存在你浏览器里，来新消息时对话自动滚到当前位置。管理员可以在界面里创建、停用、恢复、归档和删除自定义数字员工；**内置数字员工受保护，永久删除必须先归档并再次确认，历史任务和审计记录仍会保留**。

界面只读状态、不能自己改工作流——**所有会改变任务、审批或数字员工的操作都走专用后端接口，界面不能自己伪造状态，也不能执行任意系统命令**。

## 七位数字员工一览

| 数字员工 | 干什么 | 主要用到的技能 |
| --- | --- | --- |
| 秘书 | 接活、确认你要什么、分派、盯进度、管审批 | 团队排班、找技能、资料运营 |
| 数字律师 | 企业内部法务（合同/隐私/合规/用工 IP/争议分流） | 法务工作流 |
| 产品经理 | 需求、优先级、验收 | 需求澄清、PRD 起草 |
| 设计师 | 交互、视觉、原型、无障碍 | 设计、设计质检 |
| 规划师 | 架构、里程碑、依赖、风险 | 头脑风暴、写计划 |
| 研究员 | 检索、找证据、写引用 | 迭代检索、深度研究、市场调研 |
| 程序员 | 写代码、调试、测试、部署 | 编码质检、TDD、编码规范 |
| 写作 | 提纲、起草、编辑 | 写作、文案、内容策略 |

每位数字员工的页面可以看到他的介绍、能力、被调用次数、Token 消耗、成功失败记录，以及系统建议你补齐的岗位。

## 不只是这七位员工

上面这七位数字员工，是这套系统出厂时配好的「默认团队」。但 Digital Office 本质上是一个**能自己编排任意数字员工团队**的框架，不绑死任何岗位：

- **加人**：管理员可以在界面里创建新的数字员工，给他起名字、定职责、配技能、绑模型，他立刻就能被秘书调度
- **换团队**：如果你的业务用不上法务或设计，只要研究和写作，可以整套换掉团队配置——研究团队、内容团队、客服团队、垂直行业团队都行
- **自主编排**：秘书会按你定义的员工清单和各自能力，自动判断该找谁、要不要并行干、要不要等人补充信息，不需要你手动指派
- **自带约束**：不管团队怎么换，「有人负责、交代清楚、不瞎循环、关键步骤有人把关」这套机制都跟着走，不会因为你换了团队就失灵

换句话说，**这套机制装在什么团队上都成立**。当前这七位只是给你上手用的一个完整示例——你完全可以按自己的业务，配出一支只属于你的数字团队。

## 为什么任务交接更稳

每次交接都会记录：

- 你真正想做成什么、当前目标、限制和验收标准
- 哪些是事实、假设、不确定或冲突
- 证据和产物在哪里
- 已经做过哪些决定
- 哪些内容因为篇幅、权限或相关性被省略了，以及怎么找回
- 交给谁了、对方有没有核对交接印记、有没有要求补充信息

大文件只存一份，靠引用传递。详细说明见 [上下文交接规范](agent-system/docs/context-handoff.zh-CN.md)。

## 不会无限自我反思

主流 AI 实践并不存在唯一正确的「六步循环」。这个项目把干活收敛为四个能被记录、能被测试的步骤，并明确划清边界：

- AI 只产出事实、假设、依据、结论、风险、下一步动作；AI 的内心独白不保存、不展示
- 每一次干活都必须留痕：稳定编号、当前在四步的哪一步、产物、质检、卡点、可回看的事件记录、检查点和恢复点、用量和预算
- 每个副作用必须可重复、可去重或显式确认；禁止整段重来，只重做最小的可重做单元
- 暂停和恢复用持久状态和稳定编号，从检查点恢复

任务返工可以在你批准的范围和预算内自动循环。但**改数字员工、改技能、改规则、改工作流、改界面契约或改发布配置不算普通任务循环**，必须创建一个你能看到的迭代提案，由你确认。详细说明见 [LOOP 工程](agent-system/docs/loop-engineering.zh-CN.md)。

## 如果你电脑里已经有 Hermes 或个人规则

如果目标目录里已经有不是 Digital Office 管理的规则或个人数据，安装器会停下来让你明确选——**这是刻意的保护，不会替你偷偷决定**：

```bash
# 保留原来的规则，Digital Office 并排安装
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/update | bash -s -- --preserve-existing

# 备份原规则后，让 Digital Office 成为默认秘书入口
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/update | bash -s -- --overwrite-existing
```

装到 OpenClaw：

```bash
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/update | bash -s -- --host openclaw
```

管理员可以先跑两个快速检查：

```bash
~/.hermes/scripts/agent-router --health
~/.hermes/agent-system/bin/office-system health
```
