# Digital Office · 科研团队版

> 一支能查文献、定方向、做实验、写论文、审稿的 AI 科研队伍,住在你的电脑里,从立项到交付全程陪你跑。

---

## 这是什么

Digital Office 科研团队版是 Digital Office 的一个**科研专用分支**。说人话:它不是那种「你问一句它答一句」的聊天机器人,而是一支**有分工、有流程、有人盯质量**的 AI 科研队伍。

打个比方。你做科研,通常不是一个人单打独斗——得有人帮你查文献,有人帮你设计实验,有人帮你跑代码,有人帮你写论文,最后还得有人像审稿人一样挑毛病。这套系统就是把这些岗位**全都装进你的电脑**,每个岗位有一位 AI 角色负责,大家按流程接力干活,互相交接,还有人打分验收。

它跟普通 AI 工具最大的不同是:**普通 AI 你问一句它答一句,答完就散了;这套系统会记住自己干过什么、踩过什么坑、你纠正过什么,下一个项目带着经验继续干,越用越顺手。**

这套系统建立在一条朴素的信念上:**做科研最难的不是「聪明」,而是「积累」**。读过大量论文、做过很多实验、踩过坑、知道什么思路靠谱——这些经验才是科研人最值钱的东西。我们想让 AI 团队也能积累这种经验。

## 它能帮你做什么

如果你是研究生、博士、青年学者或科研助理,下面这些场景你应该很熟悉:

- **「我想研究 XXX,但不知道从哪下手」** —— 告诉科研秘书一句话,课题规划师会帮你把模糊的想法拆成具体的研究问题,定方向、找创新点、画路线图。
- **「这个领域的文献太多了,看不过来」** —— 文献研究员会帮你从 arXiv、Google Scholar、Semantic Scholar 三个源头检索,用 PRISMA 流程筛选,每条结论都追溯到原文,还能帮你找出研究空白。
- **「实验怎么设计才站得住脚」** —— 方法学专家帮你选统计方法、算样本量、做因果推断,保证你的实验有对照、有显著性、能复现。
- **「代码写完跑不出结果/跑出来不可复现」** —— 实现工程师用 MLflow 记录每次实验,Optuna 调超参,自动导出可复现包(代码+数据+配置+种子+环境),别人照着能跑出一样的数字。
- **「数据算出来不知道怎么画图」** —— 数据分析师帮你做统计检验、画出版级配图(Nature/Science/IEEE 风格)、核对每个数字。
- **「论文写完心里没底,怕被审稿人挑出来问题」** —— 质检员用顶会审稿标准模拟同行评审,把论文拆成一条条 claim 逐个验证,像 NeurIPS/ICLR 的审稿人一样找问题。
- **「怕引用了被撤稿的论文/怕自己写重了」** —— 伦理员用三个源头交叉检查撤稿标记,用 JPlag 查代码查重,用 MinHash 查文本查重。
- **「文献天天有新的,追不过来」** —— 资料员每天自动去 arXiv 等信源抓新论文,去重入库打标签,不用你手动搜。
- **「团队跑完一个项目,经验没沉淀,下次又从零开始」** —— 每次跑完都把经验沉淀下来,你纠正过的错误会记住不再犯,下一个项目带着经验干活。

## 十个科研角色

这支队伍一共 10 个角色,分三类:有些岗位每个项目都得在场,有些看情况叫来帮忙,还有一个负责后勤。

| 角色 | 干什么 | 什么时候登场 |
|---|---|---|
| 科研秘书 | 接活、分活、盯进度、评分验收 | 一直要在,所有项目都从 ta 这里开始 |
| 课题规划师(PI) | 定研究方向、找创新点、画路线图 | 一直要在,立项阶段主力 |
| 文献研究员 | 查论文、写综述、找研究空白 | 一直要在,文献阶段主力 |
| 学术写作员 | 写论文、改稿、排版 | 一直要在,写作阶段主力 |
| 方法学专家 | 设计实验、选统计方法、算样本量 | 看情况,需要实验设计时叫来 |
| 实现工程师 | 写代码、跑实验、封装可复现包 | 看情况,需要跑实验时叫来 |
| 数据分析师 | 算数据、画图、核对数字 | 看情况,需要数据分析时叫来 |
| 质检员 | 模拟同行评审、找方法学问题 | 看情况,论文初稿出来后叫来 |
| 伦理员 | 查重、查合规、查造假、查署名 | 看情况,交付前叫来做最后审查 |
| 资料员 | 定时抓新论文、自动入库、建引文网络 | 后勤,在后台默默跑 |

每个角色干完活,按交接契约把活儿交给下一个角色,链路是这样走的:

```text
科研秘书(确认要干啥,生成项目底稿)
  → 课题规划师(定方向和创新点)
  → 文献研究员(查论文,写综述,找空白)
  → 方法学专家(设计实验,定指标)
  → 实现工程师(写代码,跑实验)
  → 数据分析师(算结果,画图)
  → 学术写作员(起草论文)
  → 质检员(模拟审稿找问题)
  → 伦理员(查重查合规)
  → 交付
```

每一步都有评分或风险分,不达标就返工,返工会带反思,跑完会沉淀经验。

## 十个科研专用能力

这套系统在通用办公能力之上,叠加了 10 个科研专用 skill(技能)。下面用大白话解释每个能力是干什么的:

| 能力 | 通俗解释 |
|---|---|
| **langfuse-integration**<br>全链路可观测性 | 每次 AI 操作都有 trace 记录,就像快递的物流轨迹——你随时能查到「这一步谁干的、调了什么工具、花了多少 token、有没有失败」,不会出现「AI 说做完了但其实没做」的糊涂账。 |
| **paper-qa-rag**<br>科学文献高精度问答 | 你拿一篇论文问它问题,它不光读全文,还会检查这篇论文有没有被撤稿、跟其他论文有没有矛盾。比直接喂给 ChatGPT 准多了,因为它专门为科学文献优化过。 |
| **deep-research**<br>树状递归深度研究 | 你给一个课题,它会把问题拆成一棵树(最多 3 层),每个分支并行去查,最多搜 100 个来源,最后综合去重出一份报告。不是那种「搜一下就答」的浅尝辄止。 |
| **verification-loops**<br>多样本投票验证 | 重要结论不让 AI 一次说了算——同一个问题让 AI 独立采样 3 次,再找个独立的 judge 比对,多数一致才算数。像找三个医生会诊,比一个医生拍板靠谱。 |
| **goap-planner**<br>GOAP 目标规划 | 你说「我想研究 XXX」,它自动把这个大目标拆成一步步的小任务,找出最短路径。如果中途某步走不通,会重新规划路线,不会一条道走到黑。 |
| **reflexion-loop**<br>反思循环 | 失败了不是简单重跑,而是先反思「这次哪里不够、为什么不够、下次怎么改」,带着反思重写。有 4 种策略可选,从「不反思直接重试」到「组合反思」都支持。 |
| **storm-survey**<br>综述生成 | 借鉴斯坦福 STORM 的方法,先用多个视角(理论派/工程派/应用派/批评派)互相提问发现研究角度,再写大纲,最后写全文。关键是——**不编造引用**,每条引用都真的存在。 |
| **retraction-check**<br>撤稿专项检查 | 用三个源头(PaperQA2、Crossref、Retraction Watch)交叉检查你引用的论文有没有被撤稿。核心论点如果依赖一篇被撤稿的论文,直接打回返工。 |
| **cross-model-verification**<br>跨模型交叉验证 | 关键产出不让同一个模型家族自己审自己——找 4 个不同家族的模型分别评分,再找个第三方盲审 judge 仲裁。模型自己审自己会偏袒,跨模型才公道。 |
| **socratic-elicitation**<br>Socratic 引导立项 | 你说「我想研究 XXX」之后,它不会立刻开干,而是用苏格拉底式追问从 5 个维度探测你的真实意图:要改变什么结果?谁判断成功?交付物是什么?有什么硬限制?资料在哪?问清楚了再动手。 |

## 七道科研完整性门控

AI 做科研最容易翻车的 7 种情况,这套系统每一道都设了硬关卡。过不了关,流水线就停下来。

| 门控 | 防什么 | 通俗解释 | 失败怎么办 |
|---|---|---|---|
| **幻觉引用阻断** | 编造不存在的文献 | AI 一本正经地引用了一篇根本不存在的论文,每条引用都要通过 Semantic Scholar 验证真的存在 | 返工,最多 2 次 |
| **方法伪造阻断** | 方法步骤对不上 | 论文里写了「用了方法 X」,但找不到对应的代码脚本或文献依据 | 返工,最多 2 次 |
| **框架锁定阻断** | 跟已有工作撞车 | 你的研究框架跟近 2 年已发表的框架相似度超过 80%,还没声明差异——要么是真撞车,要么是 p-hacking | 返工,最多 2 次 |
| **数据伪造阻断** | 数字对不上原始数据 | 论文里的数字追溯不到原始数据和计算脚本,或者数字对不上——**这是伦理红线,不可商量** | **直接 fail,不可返工**,伦理员立即介入 |
| **过度泛化阻断** | claim 吹得太大 | 实验只在小样本上跑过,结论却写成「普遍适用」——claim 的范围不能超出实验覆盖的范围 | 返工,最多 2 次 |
| **撤稿来源阻断** | 依赖被撤稿的论文 | 你的核心论点依赖一篇已经被撤稿的论文——哪怕它撤稿前引用量很高 | 返工,最多 2 次 |
| **可复现性缺口阻断** | 别人复现不出来 | 没附可复现包(代码+数据+配置+种子+环境),别人照着跑不出一样的数字 | 返工,最多 2 次 |

**为什么数据伪造零容忍?** 因为其他 6 种情况可能是 AI 不小心犯的错,返工能改;但数据造假是科研伦理红线,一旦坐实,论文撤稿、声誉扫地、甚至学术生涯终结。这套系统在这一条上不留情面——宁可误杀,不可放过。

## 六维评分卡

每位角色干完活,不是 ta 自己说「我做完了」就算数,而是有一套六维评分卡打分。**满分 100 分,合格 75 分**。

| 维度 | 权重 | 评什么 | 怎么核查 | 一票否决 |
|---|---|---|---|---|
| 严谨性 | 25 分 | 方法对不对、实验够不够、引用真不真 | 工具核查引用真实性、可复现性、统计显著性 | 是 |
| 新颖性 | 20 分 | 跟已有工作有没有区别 | 跟近 2 年文献比对 | 是 |
| 清晰度 | 20 分 | 写得清不清楚、图表规不规范 | 语法检查、格式检查 | 否 |
| 可行性 | 15 分 | 资源、时间、数据够不够 | 资源评估 | 是 |
| 影响力 | 10 分 | 问题值不值得做 | — | 否 |
| 置信度 | 10 分 | 评分者自己有多确信 | — | 否 |

**什么是一票否决?** 严谨性、新颖性、可行性这三项,任何一项得分低于 50%(即该项满分的一半),即使总分过了 75,也直接打回返工。因为这三项是科研的底线——方法不严谨、思路不新颖、做不出来,其他方面再好也没用。

**评分依据是谁的标准?** NeurIPS/ICLR/ACL 顶会审稿 rubric + 斯坦福 100+ NLP 研究者的 idea 评估方法(Si et al. 2024)+ Sakana AI 的 AI-Scientist 自动评审。不是我们自己拍脑袋定的。

**评分对象是谁?** 只评两类关键产出:课题规划师的研究蓝图/idea、学术写作员的论文初稿。其他角色产出(文献综述、实验日志、数据图表)走风险分,不用评分卡。

**遇到「灰色地带」怎么办?** 新颖性和影响力这两维,有时候很难纯量化——「这个 idea 算不算撞车」「这个问题值不值得做」,评分卡只给参考分,最终判断走**人类介入决策**(下一节细说)。

## 反思进化闭环

这是这套系统最核心的设计,也是它跟普通 AI 工具拉开差距的地方。简单说就是四句话:**做得不好能返工,返工能反思,反思能沉淀,沉淀能复用。**

### 场景一:评分不合格 → 带反思返工

```text
评分不合格
  → 先生成反思("这次哪里不够、为什么不够、下次怎么改")
  → 带着反思重写(不是简单重跑,而是知道哪里错了再改)
  → 重写后重新评分
  → 同一问题连续返工 2 次 → 升级为"不要再犯"的教训,存入经验库
```

最多返工 3 轮,超了上报你决策。**关键区别:返工不是让 AI 重新掷骰子,而是让 ta 先想明白哪里错了再改。**

### 场景二:你纠正 → 记住不再犯

你说「以后查文献不要只用 Google Scholar」:

- 系统判断这是**通用偏好** → 写进文献研究员的灵魂文档,以后永远遵守
- 系统判断这是**一次性指正** → 只存当前项目,不污染长期记忆
- 系统拿不准 → 反过来问你:「这是以后都这样,还是就这次?」

### 场景三:项目跑完 → 沉淀经验

```text
项目结束
  → 回顾整个过程,抽取经验("遇到 X 情况,应该/不应该 Y")
  → 去重后存入经验库
  → 下一个项目启动时,按主题检索 top-5 经验,带着经验干活
```

### 场景四:实验失败 → 即时归档(科研特色)

科研里负面结果本身有价值——排除一个错误假设、划定一个方法的失效边界、记录一组不 work 的超参组合,都是科研产出。这套系统在**失败发生时就即时归档**,而不是等项目结束才笼统总结一句「不要 Y」。

```text
实验/假设不通过
  → 即时归档"失败模式记录":
      - 假设/方法是什么(原预期)
      - 预期结果 vs 实际结果
      - 失效条件(什么条件下不工作)
      - 可能原因(标注"推测",不冒充定论)
      - 方法边界(此方法在 X 范围内不适用)
  → 通知课题规划师评估:是否调整方向
```

### 人类介入决策(灰色地带)

不是所有判断都能靠评分卡自动裁决。这套系统设了 6 个条件,任一命中就暂停流水线,生成「决策包」请你裁决:

| 条件 | 通俗解释 |
|---|---|
| 灰色区间得分 | 新颖性/影响力得分卡在「既不到一票否决、也没过合格线」的模糊地带 |
| 低置信度 | 评分者自己都不确定,自动裁决不可信 |
| persona 分歧 | 严格派审稿人说可投、宽松派说拒,差了 2 档,自动取平均会失真 |
| claim 边界 | 结论强度和证据强度对不上,弱化到什么程度需要人定 |
| 方向性决策 | 课题规划师想换方向/放弃假设/调整核心创新点 |
| 重复返工预警 | 同一维度连续返工 2 次(还没到 3 次硬上限),提前请你介入 |

决策包里会给齐:争议点、两方观点、已尝试的自动校准及为何不够、可选方案(每个附后果估计)、秘书的建议(只建议,不替你决定)。**不是把难题原样抛给你,而是帮你把问题梳理清楚再请你拍板。**

## 与其他分支的关系

这个仓库有三个并列分支,各自面向不同场景,但共用同一套底层基础设施:

| 分支 | 定位 | 面向谁 | 特色 |
|---|---|---|---|
| **main** | 数字办公系统(通用底座) | 所有想用 AI 团队干活的人 | 通用 Agent 编排、上下文交接、质量评分、知识空间 |
| **research-team**(本分支) | 科研团队特化 | 做科研的研究生、博士、青年学者 | 10 个科研角色 + 科研完整性门控 + 文献/实验/论文全流程 |
| **writer-team** | 写作团队特化 | 做新媒体内容创作的人 | 选题、素材、撰稿、审查、去 AI 味、排版 |

**本分支跟 main 的关系**:research-team 从 main 派生,继承 main 的全部基础设施(Loop 工程、上下文交接、质量评分、知识空间、审批门控),在之上叠加 10 个科研专用角色、10 个科研专用 skill、7 道科研完整性门控、六维评分卡、反思进化闭环。你可以把 main 想象成「操作系统」,research-team 是装在操作系统上的「科研专用软件包」。

**三个分支相互学习**:2026-07 这轮迭代,三个分支互相借鉴了对方的好东西:

- research-team 吸收了 main 的 context-envelope v2.0 工作包交接规范(任务交接更稳)
- research-team 吸收了 main 的 smoke 测试体系(系统更可靠)
- research-team 吸收了 writer-team 的 Gate 模式(每道关卡必须声明「必须输出什么」和「通过条件是什么」)
- research-team 吸收了 writer-team 的 13 方法论工程化落地(把学术论文里的方法论映射到科研场景)

## 怎么用

### 第一步:装工具

大部分工具是 Python 库,一条命令装完(跟 main 共用同一个安装器,但记得切到 research-team 分支):

```bash
# 装 Digital Office 主体(切到 research-team 分支)
curl -fsSL https://raw.githubusercontent.com/digibeing1001/digital-office-agent-system/main/update | bash

# 装科研专用 Python 库
pip install arxiv scholarly semanticscholar pdfplumber \
            dowhy econml pyDOE2 statsmodels \
            mlflow optuna hydra-core stable-baselines3 \
            SciencePlots altair plotly seaborn \
            crossrefapi Pylatex latexify bibtexparser \
            factscore \
            feedparser paddleocr habanero datasketch \
            pybibliometrics knowledge-storm
```

GROBID(PDF 全文解析)和 JPlag(代码查重)需要单独部署,看各自 GitHub 仓库说明。

### 第二步:查看系统状态

```bash
# 查看某个 run 的状态(是否就绪、走到哪一步、花了多少预算、评分趋势、门控有没有触发)
python agent-system/bin/loop-status.py --run-id <run_id>

# 输出 markdown 格式(适合贴到文档里)
python agent-system/bin/loop-status.py --run-id <run_id> --markdown

# 当 CI 门控用(不就绪时返回非零退出码,适合自动化检查)
python agent-system/bin/loop-status.py --run-id <run_id> --exit-code
```

输出会告诉你:run 是否就绪、当前走到哪一步、花了多少预算还剩多少、评分趋势(在涨还是在退)、完整性门控有没有被触发、有没有需要你介入的问题。

### 第三步:跟科研秘书说一句话

直接对秘书说一件事就行,例如:

```text
帮我查一下 LLM Agent 在科研领域的最新进展,出一篇综述。
```

```text
我想研究 AI 记忆框架,帮我从立项到实验设计走一遍。
```

```text
我有一份论文初稿,帮我模拟同行评审找问题。
```

秘书不会立刻动手,会先用 Socratic 引导式追问把你的意图问清楚:

- 你要写综述、提新方法、还是复现别人的?
- 给谁看?验收标准是什么?
- 有没有硬限制(必须用国产模型?预算?截止日期?)
- 手头有没有已读的论文或草稿?

答完就分给对应角色开干。整个过程你看得见、管得住、随时能介入。
