# 多 Agent 上下文交接规范

## 目标

上下文交接的目标不是把完整聊天记录复制给下一个 Agent，而是让接收方准确知道：要完成什么、哪些事实可信、做过哪些决定、证据在哪里、缺少什么、有什么风险，以及自己是否有权执行。

正式交接由 `agent-system/context-envelope.schema.json` 和 `agent-system/context-handoff.policy.json` 共同约束。

## 三类身份

- `context_id`：同一用户意图链保持稳定。
- `task_id`：一个具体任务使用一个 id；任务进入终态后，后续修改创建新任务。
- `handoff_id`：每次交付尝试都唯一，用于去重、审计和确认。

上下文真正发生变化时，`context_version` 必须递增。

## 交接内容

交接包必须包含：

- 目标、用户意图和短摘要
- 当前 LOOP 节点和循环次数
- 限制条件与验收标准
- 已验证事实、假设、不确定项和冲突，以及各自置信度和依据
- 来源引用和产物引用
- 已做决定和依据
- 未决问题和风险
- 主动声明的省略内容及其取回方式
- 租户、项目、用户、角色和允许动作
- 上下文预算、压缩状态和状态哈希

大型文档、代码、图片和生成结果只保存一次，通过可检索引用传递。不得反复把“上一次摘要”再摘要；只要原始证据仍可访问，就保留它的引用。

## 接收确认

创建交接后状态为 `pending_acceptance`，发送方不能把它当成已经完成。接收方校验身份和 `context_hash` 后只能做三种决定：

- `accept`：内容完整，可开始工作
- `request_context`：列出缺少的字段或引用，状态变为 `needs_context`
- `reject`：说明拒绝原因

命令入口为 `handoff-create`、`handoff-ack` 和 `handoff-list`。未确认或仍缺上下文的交接会阻止 LOOP 被判定为完成。

## 禁止内容

- 私有思维链和隐藏推理轨迹
- 密码、密钥、访问令牌和凭据
- 与任务无关的其他租户数据
- 无法取回却声称存在的大型产物
- 没有标明是事实、假设还是不确定项的关键结论

## 质量指标

生产评估至少跟踪：必填字段保留率、产物可访问率、来源一致性、决策一致性、补上下文请求率、压缩比例和重复交付率。

GUI 应明确显示交接状态、上下文版本、发送方、接收方、哈希校验、缺失项、来源和产物可访问性。它不应把一段自由文本消息伪装成已经完成的任务交接。
