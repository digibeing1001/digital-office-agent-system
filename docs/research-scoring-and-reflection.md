# 科研团队:评分 + 反思 + 人类介入 + 失败归档机制

这份文档说明科研团队的每个 Agent 如何被评分、何时该请人裁决、如何反思改进、如何把失败也变成资产。它是在主系统 `research-scoring-and-reflection.md` 基础上做的增强版,补了两块科研里长期被忽略的能力:**灰色地带的人类介入决策** 和 **负面结果的价值捕获**。

## 一、为什么需要这套机制

LLM 自评会高估自己。所以:
- 评分不能只靠 LLM 内省,必须混合外部工具核查(工具增强审查范式)
- 返工不能简单重跑,必须先生成反思再带着反思重写(错误反思范式)
- 经验不能只存当前会话,必须跨项目沉淀复用(经验学习范式)
- **判断不能全交给评分卡**:科研里大量判断是灰色地带(创新性算不算撞车、claim 算不算过强、方向值不值得坚持),纯评分卡会僵化,必须设计人类介入决策(HITL)
- **失败不能只当 anti-pattern**:实验失败本身有科研价值(排除错误假设、完善方法边界),必须即时归档"什么条件下不工作",而不是等项目结束才笼统抽一句"不要 Y"

## 二、评分卡(科研专用)

### 六维评分,满分 100,合格 75

| 维度 | 权重 | 含义 | 工具核查 | 一票否决 | 灰色地带 |
|---|---|---|---|---|---|
| 严谨性 | 25 | 方法正确、实验充分、引用真实 | 引用核查 + 可复现性 + 统计显著性 | 是 | 否 |
| 新颖性 | 20 | 与已有工作的区别 | 文献比对 | 是 | **是** |
| 清晰度 | 20 | 表达清楚、图表规范 | 语法 + 格式检查 | 否 | 否 |
| 可行性 | 15 | 资源/时间/数据可得性 | 资源评估 | 是 | 否 |
| 影响力 | 10 | 问题价值 | — | 否 | **是** |
| 置信度 | 10 | 评分者自评置信度 | — | 否 | — |

**一票否决**:任一标"是"的维度得分 < 50%,即使总分过 75 也直接返工。

**灰色地带标记**:新颖性、影响力两维的判断无法纯量化(撞车边界、问题价值),评分卡只给参考分,最终判断走第三节的人类介入决策。

### 评分依据
- 学术审稿 rubric(严谨性/新颖性/清晰度/影响力)
- idea 六维评估(加可行性/兴奋度)
- AI 评审 prompt

配置文件:[scoring_config.yaml](../profiles/office-research-secretary/scoring_config.yaml)

## 三、人类介入决策(HITL)— 灰色地带组合触发

科研里不是所有判断都能靠评分卡自动裁决。主系统原来只有"返工 3 轮后上报用户决策"这一处硬性升级,太晚也太粗——很多判断在第一轮就该让人介入,而不是耗满 3 轮才上报。本分支增加一条**与返工流程并行的灰色地带主动介入通道**。

### 触发条件(任一命中即暂停流水线,生成决策包请人裁决)

| 编号 | 触发条件 | 阈值 | 典型场景 |
|---|---|---|---|
| H1 | 灰色区间得分 | 新颖性/影响力得分落在 `[50, 65)` | 既不到一票否决,也没过合格线,撞车边界模糊 |
| H2 | 低置信度 | 评分者置信度 < 60 | 评分者自己都不确定,自动裁决不可信 |
| H3 | persona 分歧 | 多 persona 评审(严格派/宽松派/领域专家派)总体判断相差 ≥ 2 档 | 一个说可投、一个说拒,自动取平均会失真 |
| H4 | claim 边界 | claim 强度与证据强度差一档且无法自动校准 | 强 claim + 中证据,弱化到什么程度需人定 |
| H5 | 方向性决策 | PI 提出换方向/放弃假设/调整核心创新点 | 涉及项目走向,不能由 Agent 自行拍板 |
| H6 | 重复返工预警 | 同一维度连续返工 = 2 次(还没到 3 次硬上限) | 再返工一次就到顶,提前请人介入避免无效循环 |

### 介入流程

```
触发任一 H 条件
  → 秘书暂停当前角色的执行(不取消,挂起)
  → 秘书生成"决策包",交用户裁决:
      1. 争议点(具体到维度/claim/假设)
      2. 两方观点(评分依据 vs 反驳依据,或 persona 之间的分歧)
      3. 已尝试的自动校准及为何不够
      4. 可选方案(每个附后果估计)
      5. 秘书的建议(只建议,不替用户决定)
  → 用户裁决
      → 裁决结果写入当前项目记忆(checkpointer),标注 [人工裁决 YYYY-MM-DD]
      → 若裁决属通用规则(如"以后这类撞车边界都按 X 处理"),由偏好内化器写入对应 Agent 的 SOUL.md
      → 流水线从挂起点恢复,带着裁决继续
```

### 自动授权阈值(减少打扰)

用户可对部分条件预设"自动授权",命中后不暂停、只记录:
- 例:用户预设"H1 灰色区间且新颖性 ≥ 55 时自动按'弱化 claim'处理"
- 配置在 `scoring_config.yaml` 的 `human_escalation.auto_authorize` 段
- 自动授权的决策仍写入项目记忆,事后可追溯

### 与"返工 3 轮硬升级"的关系

| 机制 | 触发时机 | 性质 |
|---|---|---|
| HITL 组合触发(H1–H6) | 第一轮起,任一灰色条件命中 | 主动介入,避免僵化 |
| 返工 3 轮硬升级 | 返工次数耗尽 | 兜底,防止无限循环 |

两者互补:HITL 处理"判断不清",3 轮上限处理"改不动"。命中 HITL 时不消耗返工次数;用户裁决后若仍不合格,计入返工轮次。

## 四、四类反思进化场景

### 场景 1:用户纠正 → 内化为长期偏好

用户说"以后查文献不要只用 Google Scholar"这类纠正时:

```
用户纠正
  → 偏好内化器判断:通用偏好 还是一次性指正?
    通用 → 写入该 Agent 的 SOUL.md(标日期+来源)
          格式:[用户偏好 2026-07-01] 文献检索必须先查 ACM Digital Library
    一次性 → 存入当前项目 checkpointer,不污染长期记忆
```

依据:错误反思的反思文本 + mem0 的偏好持久化

### 场景 2:秘书评分不合格 → 返工 + HITL 介入

```
秘书评分 < 75(或某维一票否决)
  → 先查第三节 HITL 触发条件
      命中 H1–H6 任一 → 走人类介入决策(不消耗返工次数)
      未命中 → 进入自动返工:
  → Reflector 生成反思文本(错误反思式)
      反思内容:哪个维度不够、为什么不够、下次怎么改
  → 反思存入 checkpointer(单会话)
  → Actor 带反思重写
  → 若同一维度连续返工 >= 2 次
      → 升级为 anti-pattern,写入经验库(跨会话)
      → 标注:"遇到 X 情况,不要 Y"
      → 同时触发 H6 预警,请人介入避免第三次无效返工
```

返工最多 3 轮,超 3 轮上报用户决策(兜底,与第三节 HITL 互补)。

### 场景 3:项目跑完 → 沉淀经验

```
项目结束
  → 经验抽取器(经验学习式)扫描整轮轨迹
  → 抽取 insights:
      成功方案 → "遇到 X 情况,应该 Y"
      失败教训 → "遇到 X 情况,不要 Y"
  → 去重(与现有经验库比对)
  → 写入 skills/experience/<agent_id>/<topic>.md
  → 更新 scoring_config(若发现某维度评分系统性偏差)
```

新项目启动时,按主题相似度检索 top-5 经验注入 Agent 的 system prompt。

### 场景 4:实验/假设失败 → 即时归档失败模式  ← 新增

科研中负面结果本身有价值:排除一个错误假设、划定一个方法的失效边界、记录一组不 work 的超参组合,都是科研产出。原来的机制只在项目结束时笼统抽一句"不要 Y",太晚太粗,失败细节早就丢了。本分支增加**失败发生时即时归档**机制,由 [failure-mode-archiving](../skills/failure-mode-archiving/SKILL.md) 技能执行。

```
实验/假设不通过(无论是否项目结束)
  → 触发角色(PI/工程师/数据分析师)调用 failure-mode-archiving
  → 即时归档"失败模式记录":
      - 假设/方法是什么(原预期)
      - 预期结果 vs 实际结果
      - 失效条件(什么条件下不工作)
      - 可能原因(标注"推测",不冒充定论)
      - 方法边界(此方法在 X 范围内不适用)
      - 是否值得后续追踪(偶发失败 vs 系统性失败)
  → 写入 skills/experience/<agent_id>/failure-modes/<topic>.md
  → 通知 PI 评估:是否调整假设/路线(若调整,触发 H5 方向性决策)
```

**与场景 3 anti-pattern 的区别**:

| | anti-pattern(场景 3) | 失败模式归档(场景 4) |
|---|---|---|
| 触发时机 | 项目结束统一抽取 | 失败发生时即时归档 |
| 粒度 | "遇到 X,不要 Y"(行为教训) | "在条件 C 下,方法 M 不工作"(方法边界) |
| 价值定位 | 避免重复犯错 | 排除错误假设,完善方法适用边界 |
| 存储位置 | `skills/experience/<agent>/` | `skills/experience/<agent>/failure-modes/` |
| 是否阻断流程 | 否(项目已结束) | 可能(若 PI 据此调整方向,触发 H5) |

两者不是二选一,都做。一次失败既会在发生时归档为失败模式,也可能在项目结束时被抽取为 anti-pattern。

## 五、三层记忆:单会话反思 / 跨会话经验 / 失败模式库

| | 单会话反思 | 跨会话经验 | 失败模式库 |
|---|---|---|---|
| 生命周期 | 一个项目内,返工轮次间 | 跨项目,永久 | 跨项目,永久,按方法/条件检索 |
| 存哪 | checkpointer / 对话上下文 | `skills/experience/<agent>/` | `skills/experience/<agent>/failure-modes/` |
| 粒度 | "这次第 2 步检索漏了 X" | "做综述类任务时,先查 survey 再查原文" | "在长上下文 > 32k 时,方法 M 的召回率掉到 X" |
| 触发 | 每次返工自动触发 | 项目结束/用户纠正时触发抽取 | 实验/假设失败时即时触发 |
| 检索方式 | 当前会话注入 | 新项目启动按主题检索 top-5 | 新实验设计前按方法/条件检索相关失败模式 |

三者都做,不二选一。

## 六、四件套存储结构

### A. SOUL.md — 性格与长期偏好
- 存:角色定位、硬规则、用户长期偏好(被纠正后内化的)、人工裁决内化的通用规则
- 改写时机:用户明确纠正后、HITL 裁决属通用规则时,由偏好内化器 append
- 只放稳定规则,不塞具体项目经验

### B. skills/experience/ — 可复用经验库
- 存:成功方案、失败教训(anti-pattern)
- 结构:`skills/experience/<agent_id>/<topic>.md`
- 新项目启动时按相似度检索 top-5 注入 prompt

### C. skills/experience/<agent>/failure-modes/ — 失败模式库  ← 新增
- 存:实验/假设失败时的即时归档记录
- 结构:`skills/experience/<agent_id>/failure-modes/<topic>.md`
- 每条记录附:来源项目 id、失效条件、方法边界、是否系统性失败
- 新实验设计前按方法/条件检索,避免重蹈覆辙

### D. scoring_config.yaml — 可调阈值
- 存:评分卡权重、合格线、返工最大轮数、HITL 触发阈值、失败归档开关
- 进化:经验回顾器定期根据历史返工率/HITL 命中率微调建议(人工确认后生效)

## 七、每个 Agent 的反思进化能力

所有 10 个角色都具备:
1. **被评分时**:接收评分卡 + 反思文本,带反思重写;命中 HITL 条件时挂起等人裁决
2. **被用户纠正时**:判断通用/一次性,内化到 SOUL.md 或 checkpointer
3. **项目结束时**:经验抽取器从轨迹抽取 insights,存入经验库
4. **新项目启动时**:按主题检索经验库 top-5,注入 system prompt
5. **实验/假设失败时**(PI/工程师/数据分析师):即时调用 failure-mode-archiving 归档失败模式  ← 新增
6. **新实验设计前**(方法学专家/工程师):检索相关失败模式,避免重蹈覆辙  ← 新增

## 八、技术栈建议

| 组件 | 推荐 | 用途 |
|---|---|---|
| 记忆骨架 | LangGraph(checkpointer + BaseStore) | 返工记忆 + 跨项目经验 + 失败模式库 |
| 偏好持久化 | SOUL.md(纳入 system prompt) | 用户纠正 + HITL 裁决内化 |
| 经验检索 | BaseStore 向量检索 或 mem0 | top-k 经验注入 + 失败模式按方法/条件检索 |
| 评分引擎 | 评分卡 + 工具增强审查 | 避免纯 LLM 自评幻觉 |
| HITL 网关 | 暂停/恢复 + 决策包生成 | 灰色地带人类介入 |
| 提示词自优化 | DSPy(可选,进阶) | 按评分自动迭代 few-shot |

## 九、关键提醒

1. **LLM 自评不可信**:评分必须混合工具核查(工具增强审查)
2. **经验库会膨胀污染**:必须做去重和衰减(经验学习的做法)
3. **SOUL.md 只放稳定规则**:别塞具体项目经验
4. **返工不是简单重跑**:必须先生成反思再重写
5. **跨会话经验需人工确认**:anti-pattern 写入经验库前建议人工 review
6. **HITL 不是甩锅**:决策包必须给齐两方观点和可选方案,不能把判断难题原样抛给用户  ← 新增
7. **失败归档不是自我开脱**:必须如实记录失效条件和可能原因,不能写成"运气不好"  ← 新增
8. **失败模式会过时**:方法迭代后旧失效条件可能不再成立,需定期复核(衰减机制同经验库)  ← 新增
