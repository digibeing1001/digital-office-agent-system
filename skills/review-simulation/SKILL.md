# review-simulation — 审稿模拟

## 用途
模拟顶会(ICLR / NeurIPS / ACL)审稿流程,按维度(严谨性 / 新颖性 / 清晰度 / 影响力)对论文草稿做审稿,产出审稿意见、各维度评分和建议,帮助作者投稿前自检。

适用于:论文投稿前自审、组内 mock review、稿件质量评估等场景。

## 触发条件
- 用户要求"模拟审稿""审稿""mock review""投稿前检查"时。
- 用户给出一篇论文草稿要求评估质量时。
- 用户提到"reviewer""审稿意见""评分"时。

## 操作步骤
1. **加载审稿 rubric**:载入审稿评分标准(各维度定义、评分区间、权重),明确顶会审稿口径。
2. **按维度审稿**:
   - 严谨性:实验设计是否合理、统计是否充分、baseline 是否公平。
   - 新颖性:方法是否真的新、与已有工作的差异是否清晰。
   - 清晰度:写作是否清楚、结构是否合理、图表是否易懂。
   - 影响力:对社区贡献大小、应用价值、可推广性。
3. **写审稿意见**:每个维度写具体意见,优点和缺点分开列,问题配证据(论文具体段落/图表)。
4. **给评分**:每维度打分(1-10)+ 置信度(1-5),给出总分。
5. **建议**:给出总体建议(接收 / 小改 / 大改 / 拒绝)及修改优先级。

## 调用示例
```
输入:论文草稿(标题/摘要/方法/实验/结论)
输出:审稿报告
```
```markdown
## 严谨性:6/10
- 问题:baseline X 用弱配置,与自家方法不公平(见第 4.2 节表 2)。
- 建议:用 baseline X 的官方推荐配置重跑。

## 新颖性:7/10
- 优点:模块 Y 的设计有新意。
- 问题:与 [Smith et al. 2023] 高度相似,需说明差异。

总体建议:大改(Major Revision)
```

## 输出格式
审稿报告,含:
- 各维度评分(1-10)+ 置信度(1-5)
- 每维度的优点 / 弱点 / 问题
- 总体建议(接收 / 小改 / 大改 / 拒绝)
- 修改建议优先级排序

## 约束
- **每条批评必须有证据**:引用论文具体段落 / 图表 / 公式,不泛泛而谈。
- 不放水,不因作者是熟人就降低标准。
- 评分要有区分度,不得每项都给中庸分。
- 建设性优先,批评同时给改进方向。
