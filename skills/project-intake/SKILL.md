# project-intake — 项目接活

## 用途
接收用户的研究需求,在正式开跑前完成意图对齐:复述确认、追问关键信息、判断是否需要降级,最终生成一份可执行的项目底稿。是整个科研流水线的入口环节,决定了后续所有角色对齐的基准。

适用于:任何新科研任务的起点。无论是"帮我做文献综述"还是"研究 X 方法在 Y 上的效果",都先走这一道。

## 触发条件
- 用户首次提出研究需求(不论粗细)
- 已有项目进入新阶段但需求发生变化
- 用户给出一段描述但缺乏关键维度信息

## 操作步骤

### 步骤 1:复述意图
1. 用自己的话把用户需求复述一遍,不超过 3 句话
2. 复述后明确问"我理解得对吗",等用户确认或修正
3. 不得在复述阶段擅自加料或缩小范围

### 步骤 2:识别模糊点
逐项检查以下维度是否清晰,缺哪项就追问哪项:
- **研究类型**:是综述 / 实证研究 / 方法复现 / 工程实现 / 调研
- **判断人**:谁有权判定"做完了"(用户本人 / 导师 / 评审 / 甲方)
- **硬边界**:时间、预算、数据范围、不可触碰的红线
- **资料起点**:用户已有啥(数据 / 文献 / 代码 / 之前的报告)
- **成功标准**:什么算"做成了",最好可验证
- **最大失败**:什么情况算彻底搞砸

### 步骤 3:追问
- 一次最多问 3 个最关键的点,不要一次抛 10 个问题吓退用户
- 用户答不上来的,标记为"待定",不要逼问
- 多轮追问直到关键维度都明确或确认走降级

### 步骤 4:判断能否降级
- 若研究类型 / 判断人 / 成功标准三项中任一项无法明确,触发降级讨论
- 降级方案:把硬研究降为软研究(如实证研究降为综述,复现降为可行性分析)
- 降级必须征得用户同意,不得擅自降级

### 步骤 5:生成项目底稿
按输出格式整理成结构化文档,作为后续所有角色的对齐基准。

## 调用示例
```
用户:帮我看看 Transformer 在长文本上的效果
复述:你想了解 Transformer 处理长文本时的性能表现,对吗?
追问:
  1. 研究类型是想做综述,还是自己跑实验对比?
  2. "长文本"指的是多长?(1万 token / 10万 token / 百万 token)
  3. 你已经有想对比的模型清单了吗?
```

## 输出格式
项目底稿(写入 `outputs/intake/<project>-<date>.md`):
- **意图**:一句话复述
- **研究类型**:综述 / 实证 / 复现 / 工程 / 调研
- **成功标准**:可验证的判定条件
- **交付物**:具体产出清单
- **硬边界**:时间 / 预算 / 数据范围 / 红线
- **最大失败**:什么情况算彻底失败
- **资料位置**:用户已有的数据 / 文献 / 代码路径
- **判断人**:谁判定完成
- **降级状态**:是否降级,降级方案

## 约束
- 不靠用户一句话就开跑,必须经过复述和追问
- 模糊就问,答不出走降级,不要硬猜
- 一次追问不超过 3 个问题
- 降级必须用户同意,不得擅自降级
- 项目底稿是后续所有角色的对齐基准,不可省略任何字段
