# reproducibility-assessment — 可复现性评估

## 用途
评估论文实验是否可复现,检查代码公开、数据公开、超参记录、随机种子、环境说明等关键信息,给出复现难度评估,缺关键信息时标"不可复现"。

适用于:论文可复现性审查、投稿前自检、复现包完整性评估等场景。

## 触发条件
- 用户要求"评估可复现性""能复现吗""复现检查"时。
- 论文投稿前需要做可复现性自检时。
- 用户提到"reproducibility""可复现""复现难度"时。

## 操作步骤
1. **检查代码公开**:是否提供代码 / 仓库链接,代码是否完整可跑,是否对应论文版本。
2. **检查数据公开**:数据集是否公开 / 可申请,split 和预处理是否说明,自建数据集是否给构建流程。
3. **检查超参记录**:学习率 / batch size / 优化器 / epoch / 正则化等是否完整列出,不是"见代码"。
4. **检查随机种子**:训练 / 评估 / 数据 shuffle 的种子是否报告,多个种子是否全列。
5. **检查环境说明**:框架版本(Torch/TF)、关键库版本、CUDA 版本、GPU 型号 / 数量、训练时长。
6. **评估复现难度**:综合以上,给出复现难度(易 / 中 / 难 / 不可复现),说明难点。

## 调用示例
```
输入:论文草稿 + 可复现包(若有)
输出:可复现性评估报告
```
```markdown
## 检查项
- 代码公开:✅ 提供仓库(github.com/xxx),版本对应
- 数据公开:⚠️ 部分公开,数据集 Y 需申请
- 超参记录:❌ 仅写"见代码",未在论文列出
- 随机种子:❌ 未报告
- 环境说明:✅ 框架版本和 GPU 已报告

## 复现难度:难
- 难点:超参未列 + 种子未报,需联系作者
```

## 输出格式
可复现性评估报告,含:
- 各检查项状态(✅ 已提供 / ⚠️ 部分提供 / ❌ 缺失)
- 复现难度(易 / 中 / 难 / 不可复现)
- 难点说明
- 修改建议(缺什么补什么)

## 约束
- **缺关键信息标记"不可复现"**:超参或种子缺失直接判"不可复现",不得标"基本可复现"。
- 不得用"见代码"代替论文中超参说明。
- 自建数据集必须给构建流程,否则标"数据不可复现"。
- 评估要站在陌生研究者视角,不假设读者有额外信息。
