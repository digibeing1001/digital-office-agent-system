# milestone-tracking — 里程碑跟踪

## 用途
对照研究路线图,跟踪各里程碑的实际完成情况,识别延期风险并上报。PI 定期执行,确保项目不悄悄滑出轨道。

适用于:项目执行期内的周期性跟踪,不用于项目启动或结题。

## 触发条件
- 定期检查(每周/每两周一次)
- 某个里程碑接近截止日
- 某个角色报告进度异常
- 用户问"项目进展怎么样"

## 操作步骤

### 步骤 1:对照路线图
1. 读取最新路线图(roadmap-planning 产出)
2. 列出所有里程碑及其计划起止时间
3. 对每个里程碑,获取当前实际状态

### 步骤 2:检查各里程碑状态
每个里程碑标注状态:
- **未开始**:还没到开始时间 / 依赖未完成
- **进行中**:已开始未完成
- **已完成**:交付物已提交且通过判断人验收
- **延期**:超过计划截止日仍未完成
- **阻塞**:因依赖/资源问题无法推进

状态判定依据:
- 已完成的判定人是路线图里指定的判断人,不是 PI 自评
- 延期判定看截止日,不看"快做完了"

### 步骤 3:识别延期风险
对"进行中"的里程碑评估延期风险:
- 进度落后:实际进度 < 应有进度(按时间比例)
- 依赖未就位:前置里程碑延期或资源未到位
- 风险事件发生:roadmap 里标注的风险点触发
- 角色报告异常:某角色卡住

延期风险分三级:
- 绿:进度正常,无风险
- 黄:进度落后但有追赶方案
- 红:已延期或必然延期

### 步骤 4:上报
- 红色里程碑:立即上报科研秘书 + 用户,附延期原因和补救方案
- 黄色里程碑:上报科研秘书,附追赶方案
- 绿色里程碑:例行记录,不上报

### 步骤 5:更新路线图
- 已完成的里程碑标记完成日期
- 延期的里程碑与用户讨论是否调整后续时间线
- 调整后续里程碑的计划时间(如前置延期,后置顺延)

## 调用示例
```
路线图检查日:Day 25
M1 文献调研 (Day 1-7):已完成 ✓ (实际 Day 8 完成,延期 1 天)
M2 方案定稿 (Day 8-14):已完成 ✓
M3 数据准备 (Day 8-21):进行中,进度 60%(应有 100%)→ 黄
M4 实验跑通 (Day 22-35):进行中,进度 10%(应有 20%)→ 黄
上报:科研秘书,M3 和 M4 进度落后,建议加派人手或延长 M4 截止日
```

## 输出格式
里程碑状态报告(写入 `outputs/pi/<project>-tracking-<date>.md`):
- **检查日期**
- **里程碑状态表**:每条含
  - 编号 / 名称
  - 计划起止时间
  - 实际起止时间(如已开始/完成)
  - 状态(未开始/进行中/已完成/延期/阻塞)
  - 延期风险(绿/黄/红)
  - 进度百分比
- **红色里程碑清单**:延期原因 + 补救方案
- **黄色里程碑清单**:追赶方案
- **路线图调整建议**:是否需要调整后续时间线

## 约束
- 延期超阈值(超过计划截止日 3 天,或进度落后 30% 以上)必须上报,不得隐瞒
- 状态判定基于事实(交付物是否提交、判断人是否验收),不基于"感觉快做完了"
- 已完成必须由路线图指定的判断人验收,PI 不得自评完成
- 路线图调整必须经用户同意,PI 不得擅自改时间线
- 上报要附方案,不能只报问题不给解
- 跟踪频率不低于每周一次,关键期(临近截止日)每日一次
