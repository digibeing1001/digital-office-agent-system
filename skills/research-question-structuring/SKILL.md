# research-question-structuring — 研究问题结构化

## 用途
把用户模糊的研究意图(往往一句话)拆解成可执行、可验证的子问题清单。PI 在项目接活后第一时间执行,产物是后续假设构建、方案设计、文献检索的输入。

适用于:任何研究项目启动阶段,意图尚未拆解成具体可回答问题之前。

## 触发条件
- project-intake 产出项目底稿后
- 用户给出新的研究意图但尚未结构化
- 已有子问题清单需要复盘/修订

## 操作步骤

### 步骤 1:解构意图
1. 从项目底稿读取"意图"字段
2. 识别意图里隐含的:研究对象(谁/什么)、研究动作(描述/比较/关联/预测/解释)、研究范围(时空/人群/领域)
3. 把意图拆成"研究对象 × 研究动作"的二维表,识别覆盖与未覆盖

### 步骤 2:识别核心变量
1. 从意图里提取核心变量:自变量( manipulated)、因变量( measured)、控制变量( controlled)、干扰变量( confounding)
2. 标注每个变量的类型:类别型 / 连续型 / 序数型
3. 标注每个变量的可测量性(可直接测量 / 需代理指标 / 不可测量)

### 步骤 3:拆子问题
1. 把意图拆成 3-7 个子问题,每个子问题对应一个可回答的"动作-对象"组合
2. 子问题类型参考:
   - 描述性问题(What is...):描述现状
   - 比较性问题(Does A differ from B...):差异检验
   - 关联性问题(Is A related to B...):相关/回归
   - 因果性问题(Does A cause B...):实验/准实验
   - 预测性问题(Can we predict B from A...):建模
3. 每个子问题标注:对应变量、问题类型、可回答性

### 步骤 4:检查 MECE
1. 检查子问题之间是否互斥(Mutually Exclusive):有重叠的合并
2. 检查子问题合起来是否穷尽(Collectively Exhaustive):意图有遗漏的补上
3. 删除"看似相关但答了也不解决意图"的子问题

### 步骤 5:定优先级
1. 按以下维度给每个子问题排序:
   - 对意图的贡献度(高/中/低)
   - 可行性(资源/时间/数据是否就位)
   - 依赖关系(是否依赖其他子问题先答)
2. 标注 P0(必须答)/ P1(应该答)/ P2(可答可不答)

## 调用示例
```
意图:研究 LoRA 微调对大模型长文本能力的影响
解构:对象=大模型,动作=比较(LoRA vs 全参),范围=长文本任务
变量:自变量=微调方式,因变量=长文本任务表现,控制=模型规模/数据
子问题:
  P0: LoRA 与全参微调在长文本任务上的性能差异有多大?
  P0: 微调方式对不同模型规模的影响是否一致?
  P1: LoRA 的秩大小如何影响长文本能力?
  P2: 不同长文本任务类型上差异是否一致?
```

## 输出格式
子问题清单(写入 `outputs/pi/<project>-questions.md`):
- **意图**:一句话
- **核心变量表**:变量名 / 类型 / 角色(自/因/控制/干扰)/ 可测量性
- **子问题清单**:每条含
  - 编号
  - 问题文本
  - 问题类型(描述/比较/关联/因果/预测)
  - 对应变量
  - 优先级(P0/P1/P2)
  - 依赖关系(依赖哪些其他子问题)
- **MECE 检查记录**:互斥/穷尽检查结果

## 约束
- 子问题必须可回答、可验证,不写"探索 X 的本质"这类不可回答的问题
- 子问题数量控制在 3-7 个,过多说明意图太大需要拆项目
- 不替方法学专家定具体方法/指标,只定"要回答什么问题"
- MECE 检查必须显式记录,不得跳过
- 变量必须标注可测量性,不可测量的需要标明走代理指标
