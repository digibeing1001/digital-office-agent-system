# 同行评审技能(Peer Review)

## 用途
在论文/报告投稿或发布前,模拟顶会审稿流程对稿件做全面审查,识别致命问题、方法学缺陷、claim 校准偏差、逻辑不一致和复现障碍,产出分级问题清单 + 修改建议 + 总体判断,帮助作者在投稿前自行修补漏洞。

适用于:论文投稿前自审、组内 mock review、申报书评审、白皮书质量把关、报告发布前审查等场景。

## 触发条件
满足以下任一条件即激活本技能:
- 用户要求"审稿""review""找问题""投稿前检查""mock review""评审"
- 用户给出一篇论文/报告草稿,要求评估质量
- 需要模拟 ICLR/NeurIPS/ACL 视角的审稿意见
- 需要对方法学做严格审查

## 操作步骤

### 步骤 1:致命伤扫描(Fatal Flaw Scan)
先快速通读,寻找"一票否决"级别的问题。发现以下任一问题立即标"致命"并进入上报流程:

1. **涉嫌造假**:
   - 数字异常完美(如所有指标都恰好优于 baseline 5%)
   - 实验结果与已知物理/数学规律矛盾
   - 引用的文献疑似不存在(用文献检索技能的验证方法核查)
   - 图表数字与正文描述严重不符
2. **claim 自相矛盾**:
   - 摘要说 A,结论说非 A
   - 方法描述与实验设置不一致(如方法说用 X,实验说用 Y)
3. **baseline 严重不公**:
   - 自家方法用强配置,baseline 用弱配置
   - 对比方法版本过时(故意挑弱版本)
   - 计算预算严重不对等且未说明
4. **复现不可能**:
   - 关键超参缺失
   - 数据集未说明 split
   - 代码/数据不公开且方法描述不足以复现
5. **学术不端迹象**:
   - 大段文字与已发表文献雷同(查重)
   - 引用堆叠自引且无必要
   - 双盲信息泄露(若应双盲)

致命问题发现后,立即停止后续步骤,标"致命-需人工核实"并上报,不得自行下结论说"作者造假"。

### 步骤 2:方法学审查(Methodological Review)
按清单逐项核查,每项标注 pass / warn / fail:

1. **baseline 公平性**:
   - 数据集是否相同?
   - 超参搜索预算是否相同?
   - 计算预算是否可比?
   - 评估协议是否一致?
2. **指标定义**:
   - 是否有明确数学定义?
   - 计算范围是否说明(全量/子集/宏微平均)?
   - 方向性是否清晰(高好还是低好)?
3. **样本量与功效**:
   - 样本量是否足以支撑显著性结论?
   - 是否报告多次运行的均值±标准差?
   - 是否做了功效分析?
4. **统计检验**:
   - 检验方法选择是否恰当(数据分布匹配)?
   - 多重比较是否做了校正?
   - 是否报告效应量和置信区间(不只 p 值)?
5. **ablation 假设**:
   - 每个 ablation 是否对应明确的科学假设?
   - 是否覆盖关键组件?
   - 假设是否可证伪?
6. **可复现性**:
   - 随机种子是否报告?
   - 软件/硬件版本是否报告?
   - 超参是否完整列出?
   - 数据预处理是否说明?
   - 代码/数据是否开源或可申请?

### 步骤 3:claim 校准(Claim Calibration)
逐条提取作者的 claim,与证据强度对比:

1. 列出所有 claim(摘要 / 引言 / 结论中的强陈述)
2. 对每条 claim 评估证据强度:
   - **强证据**:实验数据直接支撑,统计显著,多 benchmark 一致
   - **中证据**:实验数据部分支撑,但样本量不足或单一 benchmark
   - **弱证据**:无实验数据,仅基于直觉或类比
   - **无证据**:纯推测
3. 检查 claim 与证据的匹配:
   - 强 claim + 弱证据 → 标"过度声称",建议弱化措辞
   - 弱 claim + 强证据 → 标"声称不足",建议强化
4. 特别关注泛化 claim:
   - "在所有场景下都有效" → 必须有多场景实验
   - "显著优于" → 必须有统计显著性检验
   - "首次实现" → 必须查证相关工作

### 步骤 4:逻辑一致性检查(Logic Consistency)
1. **各节叙事自洽**:
   - Introduction 提出的问题是否在 Method 中被解决?
   - Method 描述的方法是否在 Results 中被评估?
   - Results 的发现是否在 Discussion 中被解读?
   - Discussion 的结论是否在 Conclusion 中被总结?
2. **方法描述与实验一致**:
   - 方法章介绍的组件是否都在实验中出现?
   - 实验用到的设置是否都在方法中描述?
3. **图表数字与正文一致**(常见翻车点):
   - 正文说"提升 5.2%",表格是不是 5.2%?
   - 正文说"在 X benchmark 上",图表是不是 X benchmark?
   - 摘要的数字是否与正文表格一致?
4. **术语一致**:
   - 同一概念是否全文统一命名?
   - 缩写是否首次定义后沿用?

### 步骤 5:复现可行性评估(Reproducibility Assessment)
模拟一位陌生研究者,仅凭论文能否复现:

1. 列出复现所需信息清单:
   - 数据集来源 / split / 预处理
   - 模型架构细节 / 初始化方式
   - 训练超参 / 优化器 / 学习率调度
   - 硬件环境 / 训练时长
   - 评估协议 / 指标计算实现
2. 逐项标注:已提供 / 部分提供 / 缺失
3. 对"缺失"项评估:能否从其他公开来源补全?
4. 总体判断:完全可复现 / 需联系作者 / 基本不可复现

### 步骤 6:审稿意见模拟(Review Simulation)
模拟三个顶会视角,给出独立的审稿意见:

1. **ICLR 视角**:侧重方法新颖性和理论洞察
   - "这个方法是否真的新?核心 insight 是什么?"
   - "理论分析是否支撑方法设计?"
2. **NeurIPS 视角**:侧重实验充分性和影响力
   - "实验是否覆盖足够多的场景?"
   - "这个工作对社区有多大影响?"
3. **ACL 视角**:侧重任务相关性和分析深度
   - "为什么选这个任务?为什么这个 benchmark?"
   - "error analysis 做了吗?失败案例有吗?"

每个视角产出:
- 优点(Strengths):3-5 条
- 弱点(Weaknesses):3-5 条
- 问题(Questions to Authors):3-5 条
- 评分(1-10)+ 置信度(1-5)

### 步骤 7:问题分级
所有发现的问题按以下分级:

| 级别 | 定义 | 处理 |
|------|------|------|
| 致命(Fatal) | 涉嫌造假 / 根本性错误 / 无法发表 | 必须解决,否则不建议投稿 |
| 重大(Major) | 方法学漏洞 / claim 过度 / 复现障碍 | 强烈建议修改后再投 |
| 中等(Moderate) | 实验不够充分 / 分析缺失 / 表述不清 | 建议修改,影响评分 |
| 小问题(Minor) | 拼写 / 格式 / 引用格式 / 措辞 | 建议修改,不影响接受 |

### 步骤 8:总体判断
基于上述审查,给出总体结论(四选一):

1. **可投稿(Ready)**:无致命/重大问题,中小问题可在 camera ready 修
2. **需小改(Minor Revision)**:无致命,有少量中等,作者修改后可投
3. **需大改(Major Revision)**:无致命,有重大问题,需补充实验/重写部分章节
4. **不通过(Reject)**:有致命问题,或重大问题过多无法在投稿前解决

## 输出格式
产出以下文件(建议放在 `outputs/review/<project>-<date>/`):
- `fatal-scan.md`:致命伤扫描结果(若有,标红上报)
- `methodological-review.md`:方法学审查清单(pass/warn/fail)
- `claim-calibration.md`:claim 与证据强度对照表
- `consistency-check.md`:逻辑一致性检查结果
- `reproducibility-assessment.md`:复现可行性评估
- `simulated-reviews.md`:三视角模拟审稿意见
- `graded-issues.md`:分级问题清单(致命/重大/中等/小问题)
- `overall-verdict.md`:总体判断 + 修改建议优先级
- `author-response-template.md`:作者回应模板(针对审稿问题)

## 约束
- **每个问题配修改建议**:不只说"这里有问题",要说"建议如何改"
- **不放水**:不因为作者是熟人或自己就降低标准
- **疑似造假标致命并上报**:不得自行定罪,但必须标记并触发人工核实流程
- **不替作者做决定**:给建议,但不擅自改写作者的 claim 或方法
- **审稿视角要多元**:不只用一个会议的标准,要覆盖多个视角
- **分级要公正**:不把小问题夸大成重大,也不把重大问题压成中等
- **保密原则**:被审稿件内容不得外泄,不得用于自己的研究
- **建设性优先**:批评要建设性,指出问题的同时给改进方向

## 依赖工具/API
- **文献核查**:literature-retrieval 技能(验证引用真实性、查相关工作)
- **实验设计核查**:experiment-design 技能(对照公平性清单)
- **写作规范核查**:academic-writing 技能(对照一致性检查)
- **查重工具**:Turnitin / iThenticate(若可用)
- **数学验证**:SymPy / 手动推导核查公式
- **代码核查**(若有开源代码):静态分析、复现实验
- **领域知识**:依赖文献研究员的分类法和综述产出
